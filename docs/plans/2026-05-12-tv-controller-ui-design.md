# TV 与手机控制端 UI 设计

日期：2026-05-12

## 背景

当前家庭 KTV 系统有两个主要前端角色：

- TV Player：当前以 HTML/Vite/React 实现，后续计划迁移为 Android 形式。
- Mobile Controller：后续仍保持网页形式，主要运行在手机上。

已确认的产品原则是：电视只负责播放与状态展示，手机负责全部点歌和控制。UI 设计需要延续这个职责边界，避免把复杂操作放到电视端。

## 已确认方向

风格选择：深色影院科技感。

目标不是夜店或舞台风，而是高端家庭客厅影音设备的感觉：

- 低亮度，适合客厅和电视长时间显示。
- OLED 近黑背景，减少电视端视觉噪音。
- 青蓝色作为主要状态光，表达科技感。
- 绿色只用于在线、可用、正在播放等正向状态。
- 交互控件保持清晰、克制、可扫读。

## 设计目标

1. TV 端以 MV/视频内容为第一视觉层，UI 只作为覆盖信息层。
2. 手机控制端优先适配手机，不为桌面密集布局牺牲单手操作。
3. 两端使用同一套视觉 token，便于后续 TV 端迁移到 Android。
4. 状态、错误、确认流程清楚，不依赖用户猜测系统当前发生了什么。
5. 保持当前业务模型：房间、当前播放、队列、搜索、在线补歌、原唱/伴唱切换。

## 视觉系统

### 颜色

| 用途 | 色值 | 说明 |
| --- | --- | --- |
| 页面背景 | `#05070D` | OLED 近黑背景 |
| 次级背景 | `#0B1020` | 深墨蓝，用于层次分隔 |
| 面板背景 | `rgba(15, 23, 42, 0.82)` | 磨砂深色面板 |
| 边框 | `rgba(148, 163, 184, 0.22)` | 低亮度描边 |
| 主文字 | `#F8FAFC` | 高对比主文字 |
| 次文字 | `#CBD5E1` | 辅助说明 |
| 弱文字 | `#94A3B8` | 标签、元信息 |
| 主强调 | `#22D3EE` | 青蓝状态光、主按钮 |
| 在线/成功 | `#34D399` | TV 在线、播放中、本地可播 |
| 警告 | `#FBBF24` | 加载、恢复、需确认 |
| 危险 | `#F87171` | 离线、冲突、播放异常 |

### 字体

- 英文标题：`Outfit`
- 英文正文：`Work Sans` 或 `Inter`
- 中文 fallback：系统黑体栈，优先使用平台默认无衬线字体

建议 CSS 字体栈：

```css
font-family: Inter, "PingFang SC", "Microsoft YaHei", system-ui, sans-serif;
```

TV 端标题可使用更几何的栈：

```css
font-family: Outfit, Inter, "PingFang SC", "Microsoft YaHei", system-ui, sans-serif;
```

### 形态

- 圆角保持在 8px 左右，避免过度圆润。
- TV 端信息栏可用轻磨砂，但不能遮挡 MV 主体。
- 手机端卡片可以有 1px 描边和轻阴影，但不做多层卡片嵌套。
- 状态光使用克制的 `box-shadow`，只用于当前播放、在线和关键主按钮。
- 不使用 emoji 图标；后续实现按钮图标时优先使用 lucide 图标。

## TV 端设计

### 总体结构

TV 端保持三层结构：

1. 视频层：全屏 MV 或待机背景。
2. 环境层：低亮度背景纹理或渐变，仅在无视频或视频黑屏时提供层次。
3. 信息层：当前状态、二维码、播放信息、错误提示。

视频播放时，UI 不应成为画面中心。只有关键状态需要短时间突出。

### 空闲页

空闲页用于“电视已连接但暂无播放”的状态。

布局：

- 左侧或中左：大标题“扫码点歌”。
- 标题下方：一句说明，例如“用手机扫码进入点歌台”。
- 右侧：大二维码，二维码保持高对比浅底。
- 底部：房间名、TV 在线状态、服务状态。

视觉：

- 背景使用深墨蓝到近黑的线性层次。
- 二维码外框使用暗色面板，不使用舞台金色。
- 青蓝状态线或小状态点用于表达“已连接”。

### 播放页

播放页以 MV 全屏为主体。

固定信息：

- 底部低亮信息栏：
  - 当前歌曲
  - 歌手
  - 播放进度
  - 当前模式：原唱 / 伴唱 / 双轨
  - 下一首
- 右上角小二维码：
  - 保留扫码入口。
  - 视觉权重低于当前播放信息。
  - 不遮挡歌词或 MV 主体区域。

交互状态：

- 加载中：底部信息栏状态变为“正在准备播放”，可有轻微青蓝动效。
- 恢复中：显示“正在恢复播放”，使用警告色但不铺满屏幕。
- 首次播放授权：居中弹层提示“点击电视开始播放”，这是浏览器限制导致的状态，后续 Android 端可移除。

### 系统状态页

包括启动、等待快照、离线、冲突和错误。

布局：

- 居中标题。
- 一句明确说明。
- 如有可操作建议，放在副文案中。

示例：

- 启动中：正在启动电视端 / 正在准备客厅电视播放环境。
- 等待中：等待房间状态 / 电视端正在等待服务端房间快照。
- 离线：电视端离线 / 请检查后台服务和电视网络。
- 冲突：已有电视端在线 / 请先关闭当前在线的电视端。

### Android 迁移约束

TV 端设计需要平台无关：

- 状态模型沿用现有 `TvDisplayState`。
- 颜色、字号、间距应抽成 token。
- 避免依赖浏览器特有的复杂 CSS 效果。
- 首次播放授权提示属于 HTML 端特例，Android 端不作为长期 UI。

## 手机控制端设计

### 信息架构

手机端只按手机优先设计。

建议主结构：

1. 顶部状态栏
2. 当前播放控制面板
3. 搜索点歌
4. 已点队列
5. 在线补歌候选
6. 确认弹层

默认入口应是点歌和当前播放控制，而不是管理后台式布局。

### 顶部状态栏

内容：

- 房间名，例如 `living-room`。
- TV 在线 / 离线状态。
- 语言切换。

设计：

- 高度紧凑。
- TV 状态使用小圆点 + 文案。
- 不占用过多首屏空间。

### 当前播放控制面板

这是手机端的核心控制区。

内容：

- 当前歌曲名。
- 歌手。
- 播放状态。
- 当前模式。
- 主操作按钮：
  - 切到原唱 / 切到伴唱
  - 切歌

设计：

- 歌曲名可以更大，支持长文本换行或截断。
- 主按钮高度不低于 48px，适合手指点击。
- 切歌为危险或警告操作，必须保留确认弹层。

### 搜索点歌

搜索是主入口。

内容：

- 搜索框：歌名 / 歌手 / 拼音 / 首字母。
- 搜索状态：搜索中、失败、无结果。
- 本地结果列表。

歌曲结果卡片：

- 歌名、歌手为主信息。
- 标签显示：本地可播、已点、版本数。
- 单版本时主按钮为“点歌”。
- 多版本时展开版本列表，每个版本显示来源、时长、清晰度、推荐标记。

优先级：

1. 本地可播结果。
2. 已点状态提示。
3. 多版本选择。
4. 在线补歌候选。

### 已点队列

队列可以作为独立区域，也可以后续改为底部 tab 中的“已点”页。

内容：

- 歌曲名、歌手。
- 可撤销删除倒计时。
- 操作：顶歌、删除、撤销。

设计：

- 不依赖滑动手势完成关键操作。
- 操作按钮要清楚可见。
- 队列为空时用简短空状态，不做营销式插画。

### 在线补歌

在线补歌是辅助功能，不应该干扰本地点歌主流程。

出现条件：

- 本地结果不足。
- API 返回在线候选。
- 用户搜索明确需要补歌。

候选卡片内容：

- 歌名、歌手。
- 来源。
- 时长。
- 类型：MV / KTV / 音频。
- 可靠度。
- 风险等级。
- 任务状态。

按钮：

- 请求补歌。
- 提交中。
- 已准备。

风险表达：

- `需复核` 和 `高风险` 使用警告/危险色。
- 不把在线候选伪装成本地可播结果。

### 弹层

现有两个确认弹层需要保留：

- 切歌确认。
- 重复点歌确认。

设计：

- 深色遮罩。
- 居中小面板。
- 主要操作和取消操作清楚区分。
- 按钮高度适合手机点击。

## 状态覆盖

本设计需要覆盖以下状态：

- TV 启动中。
- TV 等待房间快照。
- TV 空闲待点歌。
- TV 正在加载。
- TV 正在播放。
- TV 正在恢复。
- TV 离线。
- TV 设备冲突。
- HTML TV 端首次播放授权阻塞。
- 控制端连接中。
- 控制端重连中。
- 控制端错误。
- TV 在线 / 离线。
- 队列为空。
- 搜索空结果。
- 搜索失败。
- 重复点歌。
- 切歌确认。
- 在线补歌获取中 / 已准备 / 失败 / 需复核。

## 可访问性与可用性

- 手机端所有可点击目标最小高度 44px，核心操作不低于 48px。
- 深色背景下正文对比度需要满足 WCAG AA。
- 电视端远距离阅读优先，字号不随视口宽度线性缩放。
- 长歌名和长歌手名必须有换行或省略策略，不能撑破布局。
- 所有交互元素有清楚的 disabled 状态。
- 使用 `prefers-reduced-motion` 降低非必要动效。
- 错误状态不能只靠颜色表达，需要有文字说明。

## 实现边界

本设计文档只定义 UI 方向和信息架构，不直接修改实现。

后续实现时建议分阶段：

1. 建立共享 design token。
2. 改造 TV 端外观，不改变播放运行时。
3. 改造手机控制端布局和 CSS，不改变控制命令模型。
4. 补充视觉回归检查或截图检查。
5. 为未来 Android TV 端整理平台无关规范。

## 结论

本方案选择深色影院科技感，保留家庭客厅的克制和可读性，避免舞台金色和后台管理台式界面。TV 端负责沉浸展示，手机端负责高效控制，两端共享同一套状态语言和视觉 token，为后续 Android TV 端迁移留出清晰边界。
