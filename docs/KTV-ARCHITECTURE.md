# 家庭包厢式 KTV 系统设计

日期：2026-04-27

## 目标

从零设计一套适合家庭客厅场景的 KTV 系统，重点解决：

- 单房间单电视的大屏 K 歌体验
- 手机端完成全部点歌与控制
- 本地歌库与在线补歌混合使用
- 音频效果以硬件链路为主，软件不承担实时麦克风 DSP

本文只记录当前已经确认的设计结论，未确认项单独列出，避免后续讨论丢失上下文。

## 已确认的产品边界

- 场景：家庭包厢式 KTV
- 房间模型：单房间
- 播放设备：单电视大屏
- 控制方式：手机是唯一控制端，电视只负责播放画面
- 歌曲来源：混合模式
- 歌源策略：本地歌库为主，在线补歌为辅
- 音频处理：硬件为主
- 当前输出形态：电视播放
- 未来输出形态：增加一对有源音箱、麦克风、混音器

## 音频链路原则

软件系统只负责内容与控制，不负责实时人声处理。

推荐职责分工：

- 软件层负责：
  - 点歌
  - 搜索
  - 队列
  - 切歌
  - 原唱/伴唱
  - 升降调
  - 歌曲播放编排
- 硬件层负责：
  - 麦克风输入
  - 实时监听
  - 混响 / 回声
  - 基础 EQ
  - 最终人声混音输出

约束：

- 不允许软件层与硬件层同时对同一类实时人声效果重复处理
- 第一版不做实时麦克风 DSP、评分、音高分析

## 交互形态

系统分成两个前端角色：

- TV Player
  - 全屏播放
  - 展示当前歌曲、MV/字幕、下一首、二维码
  - 不承载搜索和复杂操作
- Mobile Controller
  - 搜索歌曲
  - 点歌
  - 管理已点列表
  - 切歌、删歌、顶歌
  - 查看当前播放状态

核心交互原则：

- 手机负责全部控制
- 电视只订阅服务端状态并执行播放
- 手机不直接操作电视端实现细节

## 总体架构

推荐采用：

`中心服务端 + 专用电视播放器前端 + 手机控制前端`

这是当前最适合该场景的路线，原因如下：

- 职责清晰
- 点歌与播放解耦
- 更接近真实 KTV 点歌系统
- 后续可替换电视端载体而不重做业务层

## 核心模块

### 1. KTV API

负责对手机端和电视端提供业务接口：

- 搜索歌曲
- 点歌 / 删歌 / 顶歌 / 切歌
- 查询当前播放状态
- 获取歌曲详情与可播放资源
- 提供入场二维码信息

### 2. Session Engine

这是系统的核心状态机，负责：

- 维护当前房间的唯一真相
- 当前播放歌曲
- 已点队列
- 播放进度
- 切歌原因
- 原唱 / 伴唱
- 升降调
- 播放失败重试
- 对手机端与电视端推送实时事件

### 3. Catalog Service

负责统一歌曲元数据：

- 本地歌曲与在线歌曲统一建模
- 标准化歌名、歌手、语种、版本
- 拼音、别名、搜索字段
- 去重与版本归并

### 4. Media Pipeline

负责媒体资产接入与标准化：

- 扫描本地歌库
- 解析文件名和目录规则
- 提取时长、封面、分辨率等信息
- 在线搜索候选源
- 拉取或缓存在线歌曲
- 输出统一播放资产

### 5. Player Runtime

电视端播放器运行时，负责：

- 订阅当前会话状态
- 全屏播放当前歌曲
- 预加载下一首
- 上报播放成功、失败、结束、卡顿
- 断线重连后恢复到正确状态

## 存储与部署建议

结合现有家庭服务器拓扑，建议：

- `lxc-dev`
  - 业务 API
  - 会话状态服务
  - 实时同步服务
  - 搜索服务
  - 在线歌源解析 / 下载任务
- `lxc-nas`
  - 本地 KTV 歌库
  - 在线缓存文件
  - 封面、字幕等静态媒体资源

第一版最小基础设施分工建议：

- PostgreSQL
  - 歌曲元数据
  - 歌手信息
  - 队列历史
  - 播放事件
  - 配对与控制会话信息
- NAS / 文件存储
  - 本地歌曲文件
  - 在线缓存文件
  - 封面、字幕等静态资源
- `ffmpeg / ffprobe`
  - 媒体探测
  - 编码/时长校验
  - 必要时的离线标准化

补充说明：

- `Redis`、`BullMQ`、独立 `worker` 进程不是第一版默认前提
- 若后续在线补歌、批量导入或高频 pub/sub 场景被证明需要，再作为增量基础设施引入

## 技术关键点

当前已识别的关键点如下：

### 1. 混合歌源统一模型

系统必须让“本地歌”和“在线补歌”在用户视角下尽量表现为同一种歌曲对象，否则点歌体验会割裂。

### 2. 中文搜索体验

必须支持：

- 中文歌名
- 歌手名
- 拼音全拼
- 拼音首字母
- 别名
- 繁简体

否则中文点歌体验会很差。

### 3. 播放状态一致性

所有控制都必须先落到服务端状态机，再由电视端执行。否则后续会出现状态漂移、切歌不一致、断线恢复困难等问题。

### 4. 在线歌源不稳定

在线歌源只能作为补歌层，不能作为主依赖。需要预留：

- 失效检测
- 缓存策略
- 失败回退
- 多候选源选择

### 5. 播放器可靠性

对 KTV 而言，稳定切歌和快速恢复优先级高于界面炫酷程度。播放器必须优先保证：

- 预加载
- 切歌速度
- 播放失败恢复
- 断线重连恢复

## 复杂度最高的部分

按当前判断，复杂度从高到低可先看作：

1. 混合歌源统一
2. 中文搜索与歌曲去重
3. 实时播放状态机
4. 在线源缓存与失效处理
5. 电视播放器稳定性

## 关键数据模型

第一版虽然只做单房间，但数据模型不要写死成“全局单例”。建议从一开始保留 `room` 概念，默认只有一个 `living-room`。

### 1. Artist

表示歌手实体。

核心字段建议：

- `id`
- `name`
- `normalized_name`
- `name_pinyin`
- `name_initials`
- `aliases`
- `language`

作用：

- 支撑中文名、拼音、首字母搜索
- 统一歌手别名
- 为后续热门歌手、分类页做准备

### 2. Song

表示“歌曲本身”，不直接等于某个文件。

核心字段建议：

- `id`
- `title`
- `normalized_title`
- `title_pinyin`
- `title_initials`
- `artist_id`
- `language`
- `tags`
- `release_year`
- `canonical_duration_ms`
- `search_weight`

作用：

- 作为用户点歌时看到的主对象
- 屏蔽本地文件和在线源之间的差异

### 3. SongAlias

用于补充搜索召回。

核心字段建议：

- `id`
- `song_id`
- `alias_type`（别名 / 译名 / 简繁 / 拼音变体 / 常见错字）
- `alias_text`
- `normalized_alias_text`

作用：

- 解决中文歌常见的多写法问题
- 提升模糊搜索命中率

### 4. Asset

表示“具体可播放资源”，这是整个系统最关键的拆分。

一个 `Song` 可以对应多个 `Asset`：

- 本地 `mp4`
- 在线缓存 `mp4`
- 不同分辨率版本
- 不同字幕 / 伴奏版本
- 原唱版 / 伴唱版

核心字段建议：

- `id`
- `song_id`
- `source_type`（local / online_cached / online_ephemeral）
- `asset_kind`（video / audio+lyrics / dual-track-video）
- `display_name`
- `file_path`
- `stream_url`
- `container_format`
- `video_codec`
- `audio_codec`
- `duration_ms`
- `resolution`
- `bitrate`
- `lyric_mode`（hard_sub / soft_sub / external_lrc / none）
- `vocal_mode`（original / instrumental / dual / unknown）
- `status`（ready / caching / failed / unavailable）
- `checksum`

作用：

- 让歌曲元数据与实际播放资源解耦
- 为同一首歌保留多个来源和多个版本
- 统一本地歌与在线缓存歌的播放入口

### 5. SourceRecord

记录资源来源与生命周期。

核心字段建议：

- `id`
- `asset_id`
- `provider`（本地扫描器 / 某在线源）
- `provider_song_id`
- `original_url`
- `fetch_strategy`
- `last_verified_at`
- `failure_count`
- `license_note`

作用：

- 跟踪在线源失效
- 保留资源来源证据
- 为后续换源、重拉、去重提供依据

### 6. Room

第一版只有一个房间，但模型上保留。

核心字段建议：

- `id`
- `slug`
- `name`
- `status`
- `default_player_device_id`

作用：

- 避免后续多房间时推倒状态模型
- 为播放器绑定保留接口

### 7. QueueEntry

表示一条已点歌曲记录。

核心字段建议：

- `id`
- `room_id`
- `song_id`
- `asset_id`
- `requested_by`
- `queue_position`
- `status`（queued / loading / playing / played / skipped / failed）
- `priority`
- `requested_at`
- `started_at`
- `ended_at`
- `playback_options`

其中 `playback_options` 可包含：

- 目标版本
- 是否优先伴唱
- 是否要求缓存完成后再播
- 未来扩展的升降调参数

### 8. PlaybackSession

表示当前房间的播放真相。

核心字段建议：

- `room_id`
- `current_queue_entry_id`
- `player_state`（idle / loading / playing / paused / error）
- `player_position_ms`
- `media_started_at`
- `next_queue_entry_id`
- `active_asset_id`
- `version`

作用：

- 手机端和电视端都以它为准
- `version` 用于防止并发控制乱序覆盖

### 9. DeviceSession

用于电视播放器和手机控制端接入。

核心字段建议：

- `id`
- `room_id`
- `device_type`（tv / mobile）
- `device_name`
- `last_seen_at`
- `capabilities`
- `pairing_token`

作用：

- 支撑二维码入场
- 支撑电视端心跳与重连恢复

### 10. PlaybackEvent

记录关键事件，便于排障。

核心字段建议：

- `id`
- `room_id`
- `event_type`
- `event_payload`
- `created_at`

建议记录的事件：

- 点歌
- 切歌
- 加载失败
- 缓存失败
- 播放结束
- 播放器断线 / 重连

## 数据建模原则

这里有三个原则后面不要轻易推翻：

1. `Song` 和 `Asset` 必须分离  
   用户点的是“歌”，系统播的是“资源”。

2. 第一版尽量把所有可播放内容标准化为“受控静态资源 URL”  
   不要让电视端直接依赖复杂的在线解析过程。

3. 播放状态只能由服务端状态机裁决  
   手机和电视都不是状态真相来源。

## 服务间数据流

### 1. 本地歌库接入流

流程建议：

1. `Media Pipeline` 扫描 `lxc-nas` 中的本地目录
2. 调用 `ffprobe` 提取时长、分辨率、编码等信息
3. 按命名规则和目录规则生成候选元数据
4. 归并到 `Song`
5. 为实际文件创建 `Asset`
6. 更新搜索索引

关键点：

- 本地文件命名不会永远规范，必须允许“人工修正元数据”
- 扫描器要支持增量扫描，不要每次全量重建

### 2. 在线补歌流

流程建议：

1. 手机端搜索命中本地为空或用户主动选择“在线找歌”
2. `Catalog Service` 请求在线源适配器返回候选资源
3. 用户选择具体版本
4. `Media Pipeline` 创建 `SourceRecord`
5. 启动缓存任务，将资源下载或转存到受控目录
6. 缓存完成后生成 `Asset(status=ready)`
7. 再将该 `Asset` 放入播放队列

关键原则：

- 第一版在线歌建议采用“先缓存，再播放”
- 不建议第一版就做“在线直通播放”，否则稳定性和错误处理都会变复杂

### 3. 点歌到播放流

流程建议：

1. 手机端发起点歌命令
2. `KTV API` 做参数校验
3. `Session Engine` 生成 `QueueEntry`
4. `Session Engine` 更新 `PlaybackSession`
5. 通过 WebSocket 推送最新队列和播放状态
6. 电视端收到后开始预加载或切换播放
7. 电视端回传 `loading` / `playing` / `ended` / `failed`
8. `Session Engine` 再更新全局状态并广播

关键点：

- 命令和状态更新要幂等
- 切歌、顶歌、删歌都必须落在服务端
- 电视端只执行，不自行重排队列

### 4. 播放失败恢复流

流程建议：

1. 电视端上报 `failed`
2. `Session Engine` 标记当前 `QueueEntry` 失败
3. 若同一 `Song` 存在备用 `Asset`，自动切换备用资源
4. 若无备用资源，则跳到下一首
5. 将失败事件写入 `PlaybackEvent`
6. 手机端实时提示失败原因

关键点：

- 播放失败恢复必须是产品级能力，不是简单日志
- 混合歌源系统里，备用资产切换是体验关键

## 推荐技术栈（草案）

这里给的是当前推荐，不是最终锁定。

### Monorepo

建议使用：

- `pnpm workspace`
- `Turborepo`

原因：

- 电视端、手机端、服务端都可以统一 TypeScript
- 共享类型定义、协议、校验模型会更方便

### 前端

建议：

- `React`
- `Vite`
- `TypeScript`
- `TanStack Router`
- `TanStack Query`
- `Zustand`

原因：

- TV Player 和 Mobile Controller 都是高交互前端
- Vite 比较轻，适合做两个独立 Web App
- Zustand 对播放器状态和局部 UI 状态足够直接

播放器层建议：

- 优先使用浏览器原生 `video`
- 按需接入 `hls.js`

原则：

- 第一版优先支持本地 `mp4`
- 在线资源尽量在服务端标准化，不把复杂播放协议压力留给电视端

### 后端

建议：

- `Node.js`
- `TypeScript`
- `Fastify`
- `Zod`
- `Drizzle ORM`

原因：

- Fastify 足够轻，性能和可维护性都适合实时系统
- Zod 可以作为接口和事件协议的统一校验层
- Drizzle 相对直观，适合中长期维护

实时通信建议：

- WebSocket

实现原则：

- 手机端发命令
- 服务端广播状态
- 电视端回传播放器状态

### 任务与缓存

第一版默认不预设独立任务与缓存底座。

建议：

- 先把扫描、探测、状态推进写成后端内可演进的异步模块
- 只有当导入吞吐、在线缓存或事件广播复杂度被证明超过单后端可承受范围时，再引入：
  - `Redis`
  - `BullMQ`
  - 独立 `worker`

未来若引入，它们主要用于：

- 在线补歌下载任务
- 媒体扫描任务
- 重试与失败回收
- 高扇出状态广播或热会话缓存

### 数据库与搜索

建议：

- `PostgreSQL`
- `pg_trgm`

第一版建议：

- 先不单独引入 Meilisearch / Elasticsearch
- 通过规范化字段 + 拼音字段 + `pg_trgm` 先实现中文搜索

原因：

- 第一版更重要的是搜索策略，而不是分布式搜索基础设施
- 先少一个组件，整体复杂度更低

### 媒体处理

建议：

- `ffmpeg`
- `ffprobe`

用途：

- 提取媒体元数据
- 生成缩略图
- 必要时做格式标准化

### 部署

结合现有环境建议：

- `lxc-dev` 运行 API、WebSocket、任务服务
- `lxc-nas` 提供媒体目录挂载
- 统一通过 `Caddy` 反代

推荐的应用拆分：

- `apps/tv-player`
- `apps/mobile-controller`
- `apps/api`
- `apps/worker`
- `packages/shared`
- `packages/protocol`

## 第一版最小可用范围（MVP）

第一版要解决的是“稳定可唱”，不是一次做成终极系统。

### 第一版建议必须包含

- 本地歌库扫描
- 中文搜索
- 手机点歌
- 已点队列
- 切歌 / 删歌 / 顶歌
- 电视全屏播放器
- 二维码进入控制页
- 播放状态实时同步
- 在线补歌
- 在线歌曲缓存完成后再播
- 播放失败后的基础回退

### 第一版建议尽量包含

- 最近点过
- 热门歌曲
- 简单歌手页
- 基础播放历史
- 资源健康检查

### 第一版明确不做

- 多房间
- 用户账号体系
- 软件实时麦克风效果
- 唱歌评分
- AI 人声分离
- 自动生成高质量逐字歌词
- 复杂后台 CMS
- 在线源直通播放

### 第一版建议延后评估

- 原唱 / 伴唱切换
- 升降调
- 收藏歌单
- 包厢氛围特效

延后原因：

- 这些功能并不决定“能不能先唱起来”
- 但会显著增加播放器实现复杂度和资产标准化要求

## 架构上的关键取舍

这里有几条取舍建议应当明确：

### 1. 先缓存再播放，而不是在线直通

这会牺牲一点点首播等待时间，但能换来：

- 更高稳定性
- 更统一的播放器逻辑
- 更容易做失败恢复
- 更容易做资源复用

### 2. 搜索先做准，再做花

第一版不要过度追求首页推荐算法，先把以下事情做好：

- 歌名搜得到
- 歌手搜得到
- 拼音搜得到
- 版本不混乱

### 3. 先用硬字幕 / 视频内字幕资源

如果第一版就想同时解决：

- 无字幕视频
- 外挂歌词
- 逐字高亮
- 多版本歌词对齐

项目复杂度会明显抬高。

### 4. 先保证状态机稳定，再优化 UI 细节

KTV 系统是强流程产品。

如果状态机不稳，再漂亮的 UI 都会在切歌、失败恢复、断线重连时暴露问题。

## 本地歌库目录规范

本地歌库是系统主路径，目录规范必须优先服务于：

- 稳定扫描
- 低人工维护成本
- 原唱 / 伴唱资源管理
- 中文搜索质量
- 后续在线补歌入库

### 推荐目录结构

建议采用：

```text
/ktv-library
  /songs
    /mandarin
      /周杰伦
        /七里香
          - song.json
          - cover.jpg
          - original.mp4
          - instrumental.mp4
          - lyric.lrc
  /imports
    /pending
    /needs-review
    /rejected
  /cache
    /online
  /artifacts
    /thumbnails
    /waveforms
```

目录职责：

- `songs/`：正式入库、可稳定播放的歌曲
- `imports/pending/`：新导入、尚未整理的原始资源
- `imports/needs-review/`：扫描后存在冲突或缺字段的资源
- `imports/rejected/`：确认不纳入歌库的资源
- `cache/online/`：在线补歌缓存
- `artifacts/`：扫描器或后处理生成的派生文件

### 正式歌库组织规则

建议按以下层级组织：

1. 语种
2. 主歌手
3. 主歌曲名

例如：

```text
/songs/mandarin/周杰伦/七里香/
```

原因：

- 比“按首字母”更适合中文场景
- 便于人工定位和维护
- 有利于后续做歌手页与语种分类

约束：

- 目录名使用当前系统认定的“主显示名”
- 不在目录名中堆叠过多版本标签
- 版本差异通过 `song.json` 和资产文件角色表达

### 媒体文件命名规则

建议文件名只表达“资源角色”，不表达全部业务语义。

推荐命名：

- `original.mp4`
- `instrumental.mp4`
- `mv.mp4`
- `live.mp4`
- `lyric.lrc`
- `cover.jpg`

不要推荐的命名：

- `周杰伦-七里香-国语版-1080P-伴奏-最终版2.mp4`

原因：

- 文件名承载太多信息会导致扫描器过度依赖字符串解析
- 后续人工改名、迁移和批量处理都会变脆

## `song.json` 设计

每首歌目录下建议有一个 `song.json`，作为结构化元数据入口。

### 最小字段集

第一版建议至少包含：

```json
{
  "title": "七里香",
  "artist": "周杰伦",
  "language": "mandarin",
  "aliases": ["Qi Li Xiang"],
  "searchHints": ["七里香", "qilixiang", "qlx", "周杰伦 七里香"],
  "defaultAsset": "instrumental.mp4",
  "assets": [
    {
      "file": "original.mp4",
      "vocalMode": "original"
    },
    {
      "file": "instrumental.mp4",
      "vocalMode": "instrumental"
    }
  ]
}
```

### 完整字段建议

建议完整 schema 至少支持：

- `id`
- `title`
- `artist`
- `artistDisplay`
- `language`
- `genre`
- `tags`
- `releaseYear`
- `aliases`
- `searchHints`
- `titlePinyin`
- `titleInitials`
- `artistPinyin`
- `artistInitials`
- `defaultAsset`
- `preferredVocalMode`
- `cover`
- `lyricFile`
- `status`
- `source`
- `notes`
- `assets`

其中：

- `status`：`ready / incomplete / review-required`
- `source`：说明歌曲来源，例如自采集、迁移、在线缓存转正
- `notes`：人工维护备注

### `assets` 字段建议

每个资源对象建议支持：

- `file`
- `label`
- `vocalMode`（original / instrumental / dual / unknown）
- `assetKind`（video / audio+lyrics / dual-track-video）
- `isDefault`
- `quality`
- `resolution`
- `durationMs`
- `lyricMode`
- `trackMapping`
- `status`

其中：

- `trackMapping` 用于描述后续若支持双音轨切换时的声道/音轨映射
- `status` 用于标记该资源是否可用

### 设计原则

1. `song.json` 是业务真相，文件名只做辅助
2. 扫描器允许自动补全缺失字段，但人工修正应可覆盖自动结果
3. 同一首歌允许只有一个 `Asset`，不强制必须同时具备原唱版和伴唱版

## 导入与入库流程

### 阶段划分

建议把“文件进系统”和“正式入库可点歌”拆成两个阶段。

#### 阶段一：原始导入

外部获得的资源先进入：

```text
/imports/pending/
```

允许来源：

- 手工拷贝
- 下载脚本
- 在线补歌缓存转正

此阶段不要求命名完全规范。

#### 阶段二：扫描与归类

`Media Pipeline` 扫描 `imports/pending` 后：

1. 提取媒体元数据
2. 尝试从文件名中推断歌手、歌名、版本
3. 与现有 `Song` 做模糊匹配
4. 生成候选 `song.json`
5. 判定自动入库还是进入人工复核

### 自动入库条件

满足以下条件时可自动入库：

- 歌手和歌名识别置信度高
- 未与现有歌曲发生严重冲突
- 媒体文件可正常探测
- 至少存在一个可播放资产

动作：

- 创建正式目录
- 生成 `song.json`
- 移动或复制文件到 `songs/`
- 写入数据库和搜索索引

### 进入 `needs-review` 的条件

以下情况进入人工复核：

- 文件名无法稳定识别歌手或歌名
- 同名歌曲冲突严重
- 已存在多首高相似歌曲，无法判断归属
- 资源损坏
- 时长异常
- 原唱/伴唱角色无法判断

建议在：

```text
/imports/needs-review/
```

保留原始文件和一份扫描结果摘要，便于后续修正。

### 拒绝入库条件

以下情况可直接进入 `rejected`：

- 文件不可播放
- 明确重复且质量更差
- 明显不是 KTV 可用内容
- 人工确认不纳入歌库

## 扫描器策略

### 第一层：结构化元数据优先

优先顺序建议：

1. 已存在的 `song.json`
2. 同目录下配套文件名角色
3. 目录层级信息
4. 文件名解析
5. 人工复核

这意味着：

- 正式歌库中的扫描器应尽量“相信 `song.json`”
- `imports` 中的扫描器才需要更积极地猜测和推断

### 第二层：文件名解析只做候选，不做绝对真相

文件名解析可作为初始候选来源，但不要把解析结果直接当最终元数据。

例如：

```text
周杰伦 - 七里香 (伴奏版).mp4
```

可以推断：

- artist = 周杰伦
- title = 七里香
- vocalMode = instrumental

但若后续人工修正为其他结果，应以人工结果为准。

### 第三层：增量扫描

扫描器必须支持：

- 新文件发现
- 文件修改检测
- 删除检测
- 失败重试

不要依赖全量重扫作为常态。

建议为每个资产记录：

- 文件大小
- `mtime`
- 快速校验值
- 最近一次探测结果

## 缺资源时的降级策略

### 情况 1：只有一个 `mp4`

这是第一版最常见的情况，系统必须自然支持。

处理方式：

- 允许仅存在一个 `Asset`
- `defaultAsset` 指向该资源
- 若其 `vocalMode` 不明确，则标记 `unknown`
- UI 上不展示“原唱/伴唱切换”按钮，或显示为不可用

### 情况 2：只有原唱版，没有伴唱版

处理方式：

- 正常入库
- `preferredVocalMode` 可设为 `original`
- 手机端显示“仅原唱版”

### 情况 3：只有伴唱版，没有原唱版

处理方式：

- 正常入库
- 作为可唱歌曲优先展示
- 手机端显示“仅伴唱版”

### 情况 4：原唱和伴唱都有，但时长不一致

处理方式：

- 允许入库，但标记 `review-required`
- 切换能力默认关闭
- 需要人工确认是否属于同一版本

### 情况 5：同一首歌有多个视频版本

例如：

- 官方 MV
- 演唱会版
- 纯字幕伴奏版

处理方式：

- 都作为同一 `Song` 下的不同 `Asset`
- `defaultAsset` 只选一个
- 其他版本通过“更多版本”或“指定版本点歌”暴露

## 入库与播放的关键原则

1. 不是所有导入文件都应该立刻变成可点歌曲  
   导入和正式入库必须解耦。

2. 第一版允许资源不完整，但必须可解释  
   可以缺原唱，可以缺伴唱，但状态必须明确。

3. 不要把“切换原唱/伴唱”建立在文件名猜测上  
   这类能力应建立在明确的资产建模之上。

4. 在线补歌若被转正进本地歌库，应走同一套 `song.json + assets` 结构  
   不要为在线资源单独发明第二套正式格式。

## 元数据字段分工

为了避免后续搜索、筛选、推荐和人工维护混乱，`song.json` 里的分类字段必须边界清晰。

这里先定 3 个核心字段的职责：

- `language`
- `genre`
- `tags`

原则是：

- `language` 负责语种主分类
- `genre` 负责内容风格主分类
- `tags` 负责灵活补充标签

不要让三者相互替代。

### `language`

`language` 是稳定的一层主分类，主要服务于：

- 物理目录组织
- 点歌页按语种筛选
- 搜索权重和分词策略
- 后续歌库统计

结合当前需求，第一版建议只支持：

- `mandarin`
- `cantonese`
- `other`

要求：

- 每首歌只能有一个主 `language`
- 目录中的 `<language>` 与 `song.json.language` 默认保持一致
- 若目录和元数据不一致，以 `song.json` 为准，并在扫描时提示复核

不建议放进 `language` 的信息：

- 曲风
- 情绪
- 使用场景
- 是否合唱

例如：

- `mandarin` 是正确的 `language`
- `情歌` 不是 `language`

### `genre`

`genre` 是相对稳定但可多选的内容风格分类，主要服务于：

- 首页分类入口
- 风格筛选
- 后续榜单或推荐

第一版建议用较粗的枚举，不要做得太细。

建议范围：

- `pop`
- `rock`
- `folk`
- `rap`
- `electronic`
- `soundtrack`
- `classic`
- `other`

如果一首歌确实跨风格，可支持数组，但第一版 UI 不必过度依赖它。

建议：

- 最多 1 到 3 个 `genre`
- 没把握时宁可标 `other`，不要细分过度

不建议把这些放进 `genre`：

- `80后回忆`
- `年会必唱`
- `男生高音`

这些更适合放到 `tags`。

### `tags`

`tags` 是灵活标签层，主要服务于：

- 运营型筛选
- 家庭使用场景
- 临时策展
- 个性化聚合

它不是严格枚举，可以逐步沉淀，但仍需控制数量。

适合放进 `tags` 的内容：

- `情歌`
- `对唱`
- `高音`
- `伤感`
- `聚会常唱`
- `周董热门`
- `前奏长`
- `燃`
- `适合开场`

不适合放进 `tags` 的内容：

- 主语种
- 主歌手名
- 标准曲风

这些应分别归到 `language`、`artist`、`genre`。

### 推荐字段示例

例如《七里香》可写成：

```json
{
  "language": "mandarin",
  "genre": ["pop"],
  "tags": ["情歌", "周董热门", "聚会常唱"]
}
```

例如《K歌之王》可写成：

```json
{
  "language": "cantonese",
  "genre": ["pop"],
  "tags": ["经典", "伤感", "聚会常唱"]
}
```

## 元数据使用原则

### 1. `language` 用于主导航，不用于表达细腻语义

语种字段最重要的是稳定，而不是细致。

所以第一版宁可少枚举，也不要引入：

- `国语`
- `华语`
- `中文流行`
- `普通话`

这种语义重复的写法。

### 2. `genre` 只做粗分类

第一版不要试图建立一个复杂音乐学体系。

你的系统本质是家庭 KTV，不是音乐资料库。只要足够支持：

- 浏览
- 筛选
- 轻推荐

就够了。

### 3. `tags` 要允许脏，但需要可控

标签天然会长尾化，所以建议：

- 由系统提供推荐标签候选
- 人工可增补
- 后续可以做标签归并

但第一版不要开放成完全自由输入后立刻影响主搜索排序。

### 4. 搜索排序优先级要固定

第一版建议搜索相关性优先级如下：

1. `title`
2. `artist`
3. `aliases`
4. `searchHints`
5. `tags`

原因：

- 用户点歌时，主需求是找到“明确的歌”或“明确的歌手”
- 标签只能做补充召回，不能压过主标题

### 5. 目录分类和业务分类分离

目录里只有一个 `<language>`，但业务层可以有更多维度：

- `language`
- `genre`
- `tags`
- `artist`
- `hotScore`

这意味着磁盘结构只承担“帮助维护”的作用，不承担完整产品分类职责。

## 搜索字段分工

中文 KTV 系统的搜索体验，往往不是输“完整歌名”这么简单。

用户会输入：

- 中文歌名
- 歌手名
- 拼音全拼
- 拼音首字母
- 简繁混写
- 歌手 + 歌名组合词
- 别名 / 常见错字

所以搜索字段必须分层，而不是把所有信息一股脑塞进一个全文字段。

这里先定义 6 类核心字段：

- `title`
- `normalized_title`
- `aliases`
- `searchHints`
- `title_pinyin`
- `title_initials`

歌手侧也要有对应字段：

- `artist`
- `normalized_artist`
- `artist_pinyin`
- `artist_initials`

### 1. `title`

`title` 是主显示标题，是用户看到的正式歌名。

作用：

- UI 展示
- 精确匹配的最高优先级来源
- 搜索结果主标题

要求：

- 保持自然可读
- 不做过度机器化处理
- 只保留主标题，不把版本说明堆进标题本体

例如：

- 正确：`七里香`
- 不推荐：`七里香（国语原版1080P伴奏）`

版本信息应进入 `Asset` 或独立字段，而不是污染 `title`。

### 2. `normalized_title`

`normalized_title` 是为搜索准备的标准化标题。

作用：

- 容错匹配
- 去掉符号、空格、大小写差异后的统一比较
- 去重与相似歌曲归并

建议标准化内容：

- 去除中英文括号差异
- 去除多余空格
- 统一全角 / 半角
- 统一大小写
- 统一常见符号
- 可选做简繁归一

例如：

- `七里香`
- `七里香 `
- `七里香（Live）`

在标准化后都能得到更稳定的比较基底。

原则：

- `normalized_title` 只用于匹配和比较，不直接展示给用户

### 3. `aliases`

`aliases` 是“可替代正式歌名的稳定别名集合”。

适合放入：

- 常见译名
- 常见简繁变体
- 官方副标题
- 常见中英文别称

例如：

- 《后来》如果存在常见英文别称，可放入 `aliases`
- 日文歌的罗马字名也可以放入 `aliases`

不建议放入：

- 临时搜索词
- 歌手 + 歌名拼接词
- 明显长尾的用户口语描述

原因：

- `aliases` 应保持相对稳定
- 它是“这首歌也叫这个”的语义，不是“用户可能会这么搜”

### 4. `searchHints`

`searchHints` 是“为了召回而加的辅助搜索词”。

这是最容易滥用的字段，所以要明确边界。

适合放入：

- `周杰伦 七里香`
- `七里香 周董`
- `qilixiang`
- `qlx`
- 常见错字
- 常见俗称

不适合放入：

- 全量自动生成的所有 n-gram
- 与歌曲弱相关的大量热门词
- 情绪类标签

原因：

- `searchHints` 是补召回字段，不是垃圾桶
- 一旦塞太多噪音词，排序会很快变坏

建议：

- `searchHints` 可以部分自动生成，部分人工维护
- 但需要长度和数量上限

### 5. `title_pinyin`

`title_pinyin` 用于支持中文标题拼音全拼搜索。

例如：

- `七里香 -> qilixiang`
- `不能说的秘密 -> bunengshuodemimi`

作用：

- 支撑习惯用拼音搜歌的场景
- 提供比首字母更高的可区分度

要求：

- 推荐保存为无空格、全小写的连续串
- 如果需要，也可以额外保留空格版，但主索引建议统一

### 6. `title_initials`

`title_initials` 用于拼音首字母搜索。

例如：

- `七里香 -> qlx`
- `不能说的秘密 -> bnsdmm`

作用：

- 满足快速输入习惯
- 在手机点歌场景很实用

风险：

- 首字母冲突会很多
- 所以它只能是召回辅助字段，不能作为排序主力

### 歌手侧字段

歌手侧建议采用与歌名对称的结构：

- `artist`
- `normalized_artist`
- `artist_pinyin`
- `artist_initials`

原因：

- 用户搜歌手和搜歌名一样高频
- “周杰伦”“zjl”“zhoujielun”“jaychou” 都应该自然命中

如果有明显稳定的英文艺名或别名，建议进入歌手模型的 `aliases`，而不是散落到每首歌的 `searchHints` 中。

## 搜索字段使用原则

### 1. 显示字段和检索字段必须分开

不要直接拿展示给用户的标题去承担全部搜索职责。

至少应区分：

- 人类可读字段
- 标准化比较字段
- 拼音字段
- 辅助召回字段

### 2. `aliases` 和 `searchHints` 不要混用

区别可以记成：

- `aliases` = “这首歌也叫这个”
- `searchHints` = “用户可能会这样搜”

例如：

- `Jay Chou` 更适合作为歌手别名
- `周董 七里香` 更适合作为 `searchHints`

### 3. 首字母命中要有，但权重要低

原因：

- 首字母输入很方便
- 但歧义极高

所以：

- 可以召回
- 不能轻易排在精确歌名前面

### 4. 搜索组合词要通过服务端生成

像以下组合词：

- `周杰伦 七里香`
- `七里香 周杰伦`
- `周董 七里香`

建议由服务端在索引阶段统一生成，而不是让人工逐条写入。

这样可以：

- 提高一致性
- 降低人工维护成本
- 让 `searchHints` 更可控

### 5. 不要把标签词混进标题搜索字段

例如：

- `情歌`
- `经典`
- `聚会必唱`

这些词应该走 `tags`，而不是写入 `title`、`aliases` 或 `normalized_title`。

否则会导致：

- 去重失真
- 搜索排序污染
- 数据语义混乱

## 第一版索引建议

第一版可以按“歌曲索引文档”来理解，每首歌至少准备：

- `title`
- `normalized_title`
- `title_pinyin`
- `title_initials`
- `artist`
- `normalized_artist`
- `artist_pinyin`
- `artist_initials`
- `aliases`
- `searchHints`
- `language`
- `genre`
- `tags`

### 第一版自动生成建议

可由系统自动生成：

- `normalized_title`
- `title_pinyin`
- `title_initials`
- `normalized_artist`
- `artist_pinyin`
- `artist_initials`
- 歌手 + 歌名组合型 `searchHints`

建议人工维护或人工可覆盖：

- `aliases`
- 长尾 `searchHints`
- 特殊简繁 / 译名

### 第一版搜索排序建议

建议排序大致遵循：

1. `title` 精确命中
2. `artist` 精确命中
3. `normalized_title` 高相似命中
4. `aliases` 命中
5. `title_pinyin` / `artist_pinyin` 命中
6. `title_initials` / `artist_initials` 命中
7. `searchHints` 命中
8. `tags` 补充命中

这套顺序的目的是：

- 用户明确搜歌名时，结果最稳
- 拼音和首字母好用，但不反客为主
- 标签始终只做补充

```json
}
```

## 在线歌源适配器设计

混合歌源路线里，在线歌源必须被当作“可替换的数据提供者”，而不是让业务层直接依赖某个平台的返回格式。

所以系统需要一层 `Source Adapter`。

目标：

- 屏蔽不同在线源的搜索和下载差异
- 把候选结果统一成内部模型
- 让缓存、失效检测、回退逻辑都建立在统一接口上

### 适配器职责

每个在线源适配器建议只负责：

- 搜索候选歌曲
- 获取候选资源详情
- 返回可下载或可缓存的媒体信息
- 提供最小必要的来源标识

不建议适配器直接负责：

- 队列逻辑
- 播放状态
- 搜索排序最终裁决
- 歌曲正式入库规则

这些都应该留在业务层。

### 统一接口建议

第一版建议抽象成以下 4 类能力：

#### 1. `search(query)`

输入：

- 用户搜索词
- 可选语种偏好
- 可选歌手 / 歌名结构化提示

输出统一候选结果：

- `provider`
- `providerSongId`
- `title`
- `artist`
- `language_guess`
- `duration_ms`
- `thumbnail_url`
- `source_confidence`
- `candidate_type`

其中：

- `candidate_type` 可区分 `official_mv / karaoke_video / live / unknown`
- `source_confidence` 是适配器对结果可靠性的主观置信度

#### 2. `getCandidate(providerSongId)`

输入：

- 平台侧歌曲标识

输出：

- 更完整的候选详情
- 可用分辨率
- 时长
- 描述信息
- 来源页面 URL
- 可缓存资源信息

作用：

- 避免 `search()` 返回过重信息
- 只在用户点中候选后才取详情

#### 3. `prepareFetch(candidate)`

输入：

- 候选对象

输出：

- 建议下载策略
- 预估文件大小
- 期望容器格式
- 可用下载 URL 或需要的提取参数

作用：

- 让工作队列知道该如何执行缓存
- 为后续审计和失败恢复保留上下文

#### 4. `verify(record)`

输入：

- 既有 `SourceRecord`

输出：

- 资源是否仍可拉取
- 是否需要换源
- 是否需要重新抓取

作用：

- 支撑在线缓存资源的定期健康检查

## 在线候选与本地模型对齐

在线搜索返回的不是最终 `Song`，只能算“候选来源”。

业务层要做一个 `Candidate Resolution` 步骤，把它映射到内部模型：

### 情况 1：命中已有 `Song`

如果在线候选与现有歌曲高度匹配：

- 不新建 `Song`
- 在现有 `Song` 下新增一个 `Asset` 或待缓存候选
- 复用已有搜索和展示信息

这是最理想情况。

### 情况 2：疑似已有歌，但置信度不足

如果标题和歌手高度相似，但无法完全确认：

- 先创建临时候选记录
- 等缓存完成后再做二次比对
- 默认不直接并入正式歌库

原因：

- 在线源标题可能不规范
- 同名不同版本很常见

### 情况 3：系统内不存在该歌

如果明确是新歌：

- 先创建临时 `SongDraft` 或等价记录
- 待缓存完成并通过校验后，再升级为正式 `Song`

第一版不建议用户点完在线候选就立刻创建正式永久歌曲。

## 在线缓存生命周期

在线资源必须有完整生命周期，否则歌库会越来越脏。

建议采用以下状态：

- `discovered`
- `selected`
- `fetching`
- `fetched`
- `ready`
- `failed`
- `stale`
- `promoted`
- `purged`

### 状态说明

#### `discovered`

仅表示搜索时发现候选，还没有被用户真正点中。

此状态通常只保留轻量缓存，不落地大文件。

#### `selected`

表示用户已经选中这个在线候选，系统准备进入缓存流程。

此时应创建：

- 候选记录
- 初始 `SourceRecord`
- 任务上下文

#### `fetching`

后台任务正在拉取或转存资源。

要求：

- 记录开始时间
- 记录当前重试次数
- 记录任务归属的 `QueueEntry`

#### `fetched`

资源文件已经拿到，但还没完成完整校验和标准化。

此阶段可做：

- `ffprobe`
- 文件校验
- 格式判断
- 缩略图生成

#### `ready`

表示资源已经成为可播 `Asset`。

只有到这个状态，才能被加入实际播放链路。

#### `failed`

表示当前缓存流程失败。

失败应附带：

- 失败阶段
- 错误码
- 可否重试
- 最后一次错误信息

#### `stale`

表示这个在线缓存资源可能已过时，例如：

- 来源长期不可验证
- 文件损坏
- 平台策略变化

`stale` 不一定立刻删除，但不应再作为优先资源。

#### `promoted`

表示该资源已被人工或策略性“转正”为正式本地歌库的一部分。

一旦进入 `promoted`：

- 应迁移到 `songs/` 正式目录结构
- 生成或完善 `song.json`
- 不再按临时在线缓存对待

#### `purged`

表示缓存被清理，不再保留文件实体。

可保留最小化的来源记录，便于后续再拉。

## 缓存策略建议

### 1. 首播必须完成缓存

第一版在线歌建议：

- 用户选择在线候选后，先进入缓存队列
- 缓存完成才允许进入电视播放链路

这能保持：

- 播放器逻辑统一
- 失败处理统一
- 电视端实现简单

### 2. 缓存目录与正式歌库分离

建议在线缓存放在：

```text
/ktv-library/cache/online/
```

而不是直接写进正式歌库。

原因：

- 缓存资源的元数据通常不够稳定
- 需要单独清理策略
- 不应污染正式歌库目录

### 3. 缓存保留策略

第一版建议采用简单策略：

- 最近播放过的在线歌曲保留
- 长期未播放且未转正的可清理
- 失败缓存自动清理

建议至少记录：

- `last_played_at`
- `play_count`
- `promoted_at`
- `purged_at`

### 4. 同歌多缓存去重

如果同一在线候选被多次点中：

- 应优先复用已有 `ready` 资源
- 不要重复下载

若同一 `Song` 存在多个在线来源版本：

- 允许并存
- 但应有明确优先级

## 在线缓存转正策略

“转正”指的是把在线获取的资源变成正式本地歌库的一部分。

### 什么时候适合转正

建议满足以下任一条件时，进入人工转正候选：

- 被重复点唱多次
- 明显质量优于现有本地资源
- 系统内原本没有该歌
- 用户主动标记“加入正式歌库”

### 转正流程建议

1. 选择某个 `ready` 状态在线资源
2. 人工确认歌手、歌名、语种、版本
3. 生成正式目录：
   - `/songs/<language>/<artist>/<title>/`
4. 移动或复制媒体文件
5. 生成或补全 `song.json`
6. 将对应缓存记录状态改为 `promoted`
7. 更新数据库中的正式 `Song` 与 `Asset`

### 不建议自动转正的原因

第一版不建议“只要播过就自动变正式歌库”，原因是：

- 在线源元数据质量不稳定
- 容易把错误版本和低质量资源沉淀进歌库
- 长期会污染正式目录结构

所以：

- 自动缓存可以做
- 自动转正不要做

## 在线歌源的失败回退策略

### 1. 搜索阶段失败

处理方式：

- 在线源失败不影响本地搜索结果
- UI 上提示“在线补歌暂不可用”
- 不阻塞本地点歌主流程

### 2. 缓存阶段失败

处理方式：

- 标记候选为 `failed`
- 若该 `Song` 存在本地备用 `Asset`，自动回退到本地资源
- 若无备用资源，则提示用户本次补歌失败

### 3. 播放前校验失败

处理方式：

- 资源不进入 `ready`
- 尝试切换同源其他分辨率或备选候选
- 仍失败则跳回“重新选版本”

## 在线歌源设计原则

1. 在线源是补充层，不是核心真相层  
   核心真相仍然是 `Song`、`Asset`、`PlaybackSession`。

2. 电视端永远不直接理解在线平台  
   电视只处理受控可播资源 URL。

3. 在线候选先做“来源记录”，再做“正式资产”  
   不要跳过缓存生命周期。

4. 转正必须显式发生  
   缓存文件和正式歌库要有清晰边界。

5. 第一版宁可保守，也不要追求在线源过度自动化  
   在线歌源是最容易失控和腐化的部分。

## API 与实时协议设计

这一层的目标只有一个：

- 手机端负责发命令
- 电视端负责执行播放并上报事实
- 服务端负责裁决状态并广播真相

不要让任何客户端绕过服务端直接定义“当前播放状态”。

## 协议设计原则

### 1. 命令、状态、事件必须分开

建议明确区分 3 类对象：

- `Command`
- `State`
- `Event`

定义：

- `Command`：客户端请求系统做某件事
- `State`：系统当前真相
- `Event`：某件事已经发生的记录或通知

例如：

- “点这首歌”是 `Command`
- “当前正在播放七里香”是 `State`
- “播放器已开始播放七里香”是 `Event`

### 2. 所有命令先进入服务端

手机端不能直接决定：

- 当前播放哪首
- 队列顺序
- 播放失败如何处理

这些都必须先由服务端状态机裁决。

### 3. 电视端只报告事实，不下业务判断

电视端应上报：

- 已开始加载
- 已开始播放
- 已播放结束
- 加载失败

但电视端不应自行决定：

- 失败后切哪首
- 是否删除队列项
- 是否切备用资源

### 4. 服务端广播的是完整状态快照 + 关键事件

第一版不要只依赖“增量 patch”，否则断线恢复会更麻烦。

建议：

- 关键变更时广播完整 `PlaybackSession + QueueSummary`
- 同时附带轻量事件流

这样：

- 新加入的手机端容易追上当前状态
- 电视端重连后也更容易恢复

## API 分层建议

建议分成两层：

- HTTP API：查询与显式命令入口
- WebSocket：实时状态同步与播放器上报

### HTTP API 适合承载

- 搜索歌曲
- 获取歌曲详情
- 获取队列快照
- 获取当前房间状态
- 提交点歌 / 切歌 / 删歌 / 顶歌命令
- 申请二维码配对信息

### WebSocket 适合承载

- 电视端心跳
- 电视端播放器状态上报
- 服务端向手机和电视推送状态变更
- 队列更新广播
- 当前歌曲切换通知

## HTTP API 草案

以下是第一版建议的主要接口。

### 查询类

#### `GET /api/rooms/:roomId/state`

返回当前房间完整播放状态：

- 当前 `PlaybackSession`
- 当前播放歌曲摘要
- 当前队列摘要
- 播放器在线状态

用途：

- 手机端首次进入时拉取完整状态
- 电视端重连后做一次全量同步

#### `GET /api/rooms/:roomId/queue`

返回房间当前已点队列。

建议返回：

- `queued`
- `loading`
- `playing`
- `up-next`

避免客户端自行拼接状态。

#### `GET /api/songs/search?q=...`

搜索歌曲。

可选参数建议：

- `q`
- `language`
- `limit`
- `includeOnline`

返回：

- 本地命中结果
- 可选的在线候选结果

#### `GET /api/songs/:songId`

返回歌曲详情。

建议包含：

- 基础元数据
- 可用 `Asset`
- 默认播放资源
- 是否存在原唱 / 伴唱

### 命令类

#### `POST /api/rooms/:roomId/commands/queue-song`

请求将歌曲加入队列。

建议请求体：

- `songId`
- `assetId`（可选）
- `requestedBy`
- `playbackOptions`

返回：

- 新的 `queueEntryId`
- 当前队列摘要
- 当前状态版本号

#### `POST /api/rooms/:roomId/commands/queue-online-candidate`

请求将在线候选加入“缓存并待播”流程。

请求体建议：

- `provider`
- `providerSongId`
- `requestedBy`
- `preferredCandidateMeta`

返回：

- 候选任务 ID
- 当前缓存任务状态

说明：

- 第一版不直接返回“已入队播放”
- 需要先经过缓存完成，再转入正式 `QueueEntry`

#### `POST /api/rooms/:roomId/commands/skip-current`

请求切到下一首。

建议请求体：

- `requestedBy`
- `reason`

#### `POST /api/rooms/:roomId/commands/remove-queue-entry`

删除尚未播放的队列项。

请求体建议：

- `queueEntryId`
- `requestedBy`

#### `POST /api/rooms/:roomId/commands/move-queue-entry`

用于顶歌或调整顺序。

请求体建议：

- `queueEntryId`
- `targetPosition`
- `requestedBy`

### 配对类

#### `GET /api/rooms/:roomId/pairing`

返回当前房间二维码接入信息。

建议包含：

- 控制端 URL
- `pairingToken`
- 过期时间

## WebSocket 协议草案

建议统一采用带 `type` 的消息 envelope：

```json
{
  "type": "session.state",
  "roomId": "living-room",
  "version": 42,
  "payload": {}
}
```

字段建议：

- `type`
- `roomId`
- `version`
- `timestamp`
- `payload`

### 手机端订阅消息

手机端主要接收服务端广播。

建议至少支持：

#### `session.state`

完整会话状态快照。

建议内容：

- 当前 `PlaybackSession`
- 当前歌曲摘要
- 队列摘要
- TV 在线状态

#### `queue.updated`

队列发生变化时推送。

建议内容：

- 队列列表
- 变化原因
- 影响的 `queueEntryId`

#### `player.updated`

播放器状态变化时推送。

建议内容：

- `idle / loading / playing / paused / error`
- 当前资源
- 当前进度

#### `candidate.updated`

在线候选缓存任务状态变化。

建议内容：

- 任务 ID
- 候选 ID
- 当前阶段
- 是否已可转入播放队列

#### `system.notice`

系统提示。

例如：

- 在线补歌失败
- 当前歌曲已跳过
- TV 端断线重连

### 电视端上报消息

电视端除了接收状态，也需要向服务端上报播放器事实。

建议至少支持：

#### `player.hello`

电视端建立连接时发送。

建议内容：

- `deviceId`
- `deviceName`
- `appVersion`
- `capabilities`

作用：

- 完成设备识别
- 让服务端知道当前连接的是哪个播放器

#### `player.heartbeat`

定期心跳。

建议内容：

- `deviceId`
- `currentQueueEntryId`
- `playbackPositionMs`
- `health`

作用：

- 判断电视端是否在线
- 为重连恢复提供最后已知进度

#### `player.loading`

电视端开始加载资源。

建议内容：

- `queueEntryId`
- `assetId`

#### `player.playing`

电视端实际开始播放。

建议内容：

- `queueEntryId`
- `assetId`
- `startedAt`

#### `player.ended`

电视端完成播放。

建议内容：

- `queueEntryId`
- `assetId`
- `endedReason`（finished / skipped / interrupted）

#### `player.failed`

电视端播放失败。

建议内容：

- `queueEntryId`
- `assetId`
- `stage`（load / decode / network / runtime）
- `errorCode`
- `message`

### 服务端下发给电视端的核心消息

建议至少包含：

#### `playback.target`

这是电视端最重要的指令型状态消息。

内容建议：

- 当前应播放的 `queueEntryId`
- 应使用的 `asset`
- 是否应立即切换
- 是否预加载下一首

作用：

- 电视端不需要理解业务，只需要执行“目标播放状态”

#### `session.sync`

全量同步消息。

适用于：

- 电视端初次连接
- 电视端重连
- 服务端发现版本差距过大

#### `player.stop`

明确通知播放器停止当前资源。

适用于：

- 队列被清空
- 当前歌曲被移除
- 状态机进入错误恢复流程

## Session Engine 状态机设计

这是系统最重要的部分之一。

第一版建议先把会话状态机控制在较小集合中。

### 房间播放状态

建议主状态只有：

- `idle`
- `preparing`
- `loading`
- `playing`
- `recovering`
- `error`

定义：

#### `idle`

当前无歌播放，队列为空或尚未触发下一首。

#### `preparing`

服务端已经选定下一首，但资源仍在准备：

- 本地资源校验
- 在线缓存等待完成
- 备用资源切换

#### `loading`

电视端已收到目标资源，正在加载中。

#### `playing`

电视端已经上报开始播放。

#### `recovering`

当前歌曲出现故障，系统正在尝试：

- 切换备用 `Asset`
- 跳过到下一首
- 回滚到稳定状态

#### `error`

当前房间进入不可自动恢复的错误状态。

第一版尽量避免频繁进入该状态。

### 状态流转建议

主流程建议：

```text
idle
  -> preparing
  -> loading
  -> playing
  -> idle / preparing
```

故障流程建议：

```text
loading / playing
  -> recovering
  -> loading / playing / idle / error
```

### 队列项状态

建议每个 `QueueEntry` 至少支持：

- `queued`
- `preparing`
- `loading`
- `playing`
- `played`
- `skipped`
- `failed`

说明：

- 房间状态和队列项状态不要混用
- 队列项描述“这首歌的生命周期”
- 房间状态描述“播放器当前整体状态”

## 状态机关键规则

### 1. 只有一个当前播放项

单房间单电视场景下，任意时刻只能有一个：

- `currentQueueEntryId`
- `activeAssetId`

这两个字段必须由服务端独占维护。

### 2. 命令必须带版本感知

第一版建议所有写命令都返回最新 `version`。

必要时客户端提交命令时可附带：

- `expectedVersion`

作用：

- 避免多个手机同时操作时发生静默覆盖

### 3. 结束事件必须幂等

电视端可能重复上报：

- `playing`
- `ended`
- `failed`

服务端必须允许幂等处理，不能因为重复事件导致队列推进两次。

### 4. 服务端负责推进下一首

当当前歌曲结束后：

- 电视端只上报 `ended`
- 服务端负责决定下一首是什么
- 服务端再广播新的 `playback.target`

### 5. 在线候选未 `ready` 前不能进入真正播放状态

可以进入：

- 候选任务队列
- `preparing`

但不能直接进入：

- `loading`
- `playing`

否则播放器会被不稳定外部源拖垮。

## 第一版最关键的协议约束

1. 手机端发命令，不发真相  
   手机端是控制器，不是状态源。

2. 电视端报事实，不报决策  
   电视端是播放器，不是调度器。

3. 服务端产出完整状态  
   所有客户端都以服务端状态为准。

4. 在线补歌先变成 `ready asset` 再进入播放链  
   不让在线源直接污染播放协议。

5. 状态流转要能断线恢复  
   新连接客户端必须能靠全量状态快照重新进入正确画面。

## 歌词 / 字幕策略

歌词与字幕是 KTV 体验核心，但也是最容易把第一版复杂度拉爆的部分。

这里先明确一个总原则：

- 第一版优先解决“稳定看得到歌词并能唱”
- 不优先解决“最华丽的逐字高亮歌词引擎”

## 字幕形态分层

建议把可用资源分成 4 类：

### 1. `hard_sub`

歌词已经烧录在视频画面里。

特点：

- 播放最简单
- 兼容性最好
- 不依赖额外字幕解析
- 无法做样式改造和逐字高亮增强

适用结论：

- 第一版主推
- 本地歌库和在线缓存都优先接受这类资源

### 2. `soft_sub`

视频内嵌字幕轨或外挂标准字幕轨。

特点：

- 可切换 / 可隐藏
- 理论上更灵活
- 但播放器实现复杂度更高
- 字幕样式、时序、兼容性问题会更多

适用结论：

- 第一版可识别，但不作为主路径
- 若浏览器原生支持稳定，可作为兼容增强

### 3. `external_lrc`

外部 `lrc` 或等价歌词文件。

特点：

- 适合音频型资源
- 更容易做逐句或逐字高亮
- 但歌词质量、时间轴质量、版本匹配问题会很重

适用结论：

- 第一版只做数据模型预留
- 不作为第一版主播放体验依赖

### 4. `none`

没有可用歌词或字幕。

适用结论：

- 第一版不建议作为正式 KTV 歌库主资源
- 可存在于导入区或候选区
- 只有在用户明确接受时才允许以“无字幕资源”播放

## 第一版资源准入策略

第一版建议把资源准入规则写得保守一点。

### 正式歌库主路径

正式歌库中的可点歌曲，优先要求：

- 视频资源可正常播放
- 至少存在可见歌词

优先级建议：

1. `hard_sub`
2. `soft_sub`
3. `external_lrc`
4. `none`

### 第一版推荐准入标准

建议：

- `hard_sub`：直接视为第一优先正式资源
- `soft_sub`：可入库，但需要确认在目标播放器上稳定
- `external_lrc`：先支持建模，不保证第一版完整体验
- `none`：默认不进正式歌库，除非人工特批

原因：

- 你要做的是“先能稳定唱起来”
- 不是先做一个复杂歌词引擎

## `Asset.lyricMode` 的职责

前面 `Asset` 已经预留了 `lyricMode`，这里把语义定死：

- `hard_sub`
- `soft_sub`
- `external_lrc`
- `none`

它的作用是：

- 告诉播放器该如何渲染歌词
- 告诉扫描器该如何判断资源完整性
- 告诉点歌系统这首歌的歌词体验等级

建议同时补一个辅助字段：

- `lyricQuality`

可选值建议：

- `verified`
- `usable`
- `poor`
- `unknown`

作用：

- 区分“有歌词”和“歌词质量真的可唱”不是一回事

## 第一版播放器歌词能力边界

### 必须支持

- 播放带 `hard_sub` 的视频
- 在 UI 上正确展示该资源是否支持原唱 / 伴唱、歌词类型等基础信息
- 对 `none` 资源做明确降级提示

### 尽量支持

- 对 `soft_sub` 资源做有限支持
- 优先依赖浏览器原生视频字幕能力

### 明确不做

- 自研复杂逐字高亮引擎
- 自动歌词对齐
- 自动从音频生成歌词时间轴
- 多版本歌词智能匹配
- 演唱评分级别的音节同步

## 扫描器的歌词识别策略

扫描器需要回答两个问题：

1. 这首资源有没有歌词
2. 这份歌词靠不靠谱

### 对 `hard_sub`

第一版不要求自动 OCR 检测歌词内容。

处理建议：

- 主要依赖人工标注或来源可信度
- `song.json` 或导入规则可直接声明该资源为 `hard_sub`

原因：

- 自动识别视频里是否有硬字幕，准确率和实现成本都不划算

### 对 `soft_sub`

扫描器可检测：

- 是否存在字幕轨
- 字幕轨语言
- 字幕轨数量

但第一版不要求做复杂字幕内容质量分析。

### 对 `external_lrc`

扫描器可检测：

- 文件是否存在
- 编码是否可解析
- 时间轴格式是否基本合法

如果检测到：

- 无时间标签
- 大量空行
- 与资源时长差异过大

则标记为 `poor` 或 `review-required`。

## 歌词与资源版本匹配原则

一首歌可能有：

- 原唱版视频
- 伴唱版视频
- 单独的 `lrc`
- 现场版

这些歌词未必能混用。

所以建议：

1. 歌词绑定到 `Asset`，不要只绑定到 `Song`
2. 若一个 `lrc` 只适用于某个版本，就挂在该 `Asset` 上
3. 不要假设同一首歌的所有版本都共用同一份歌词

这是后面避免字幕错位的关键。

## 第一版具体策略建议

### 正式歌库

优先收录：

- 已带硬字幕的中文 KTV 视频

这是最符合你当前目标的主路线。

### 在线补歌

优先挑选：

- 已带清晰硬字幕的 KTV / 伴奏视频

不建议第一版为了补歌去接：

- 只有音频无歌词
- 只有普通 MV 无适合演唱字幕

### 控制端展示

手机端可给每首歌显示轻量标识：

- `硬字幕`
- `软字幕`
- `外部歌词`
- `无歌词`

这样用户在点歌时心里有数。

## 歌词 / 字幕策略的关键取舍

### 1. 第一版优先硬字幕

这是最重要的取舍。

它牺牲了：

- UI 深度可定制性
- 逐字高亮高级效果

但换来了：

- 最低的播放器复杂度
- 最稳定的首版体验
- 最低的歌词错位风险

### 2. `external_lrc` 先做结构，不做重体验

这意味着：

- 数据模型先支持
- 以后可以扩
- 但第一版不围绕它设计主流程

### 3. 歌词质量要有显式状态

不要把“有个 `lrc` 文件”直接等同于“这首歌可唱”。

系统必须能区分：

- 有歌词
- 歌词可用
- 歌词高质量

## 这部分对后续设计的影响

这套策略会直接影响后续几件事：

- 原唱 / 伴唱切换更适合建立在“双视频资产”或“明确音轨资产”之上
- 升降调若第一版做，不能依赖复杂歌词引擎
- 在线候选筛选要把“是否有清晰可唱字幕”作为重要质量因子
- 第一版正式歌库应优先沉淀“硬字幕 KTV 视频”

## 原唱 / 伴唱切换策略

原唱 / 伴唱切换是典型的“用户很想要，但实现路径差异极大”的功能。

这里先定一个总原则：

- 第一版如果做，只做“明确资产之间的切换”
- 不做依赖复杂实时音频处理的智能切换

也就是说，切换能力应建立在“歌库里本来就有可切换资源”之上，而不是运行时猜测或处理。

## 可行实现路径对比

### 方案 1：双视频资产切换

同一首歌准备两个独立视频资源：

- `original.mp4`
- `instrumental.mp4`

优点：

- 模型清晰
- 扫描器容易理解
- 播放器实现最直接
- 最适合你当前“硬字幕 KTV 视频”为主的路线

缺点：

- 需要维护两份资源
- 两份视频必须尽量同版本、同时长，否则切换体验差

结论：

- 第一版推荐主方案

### 方案 2：单视频双音轨切换

一个视频文件内含：

- 原唱音轨
- 伴唱音轨

优点：

- 理论上切换更自然
- 资源组织更优雅

缺点：

- 歌源难获取
- 浏览器和封装格式兼容性更复杂
- 扫描器要理解轨道映射
- 电视端播放器实现复杂度明显提高

结论：

- 适合作为后续增强
- 第一版不作为主路径

### 方案 3：软件实时做人声消除

只有原始音轨，运行时做伴奏分离或人声抑制。

优点：

- 理论上歌源最自由

缺点：

- 复杂度极高
- 效果不稳定
- 对浏览器端和 CPU 都不友好
- 家庭 KTV 场景很容易翻车

结论：

- 第一版明确不做

## 第一版建议方案

第一版建议采用：

- 主方案：`双视频资产切换`
- 运行时目标：正式歌库主路径支持播放中原唱 / 伴唱切换
- 降级策略：未成对资源可保留为候选或补歌候选，但不进入正式可点主路径

也就是：

- 如果歌有 `original.mp4 + instrumental.mp4` 且属于同一 `switchFamily`，系统应支持播放中切换
- 如果只有一个资源，或配对资源无法证明属于同版本同时间轴，则不能把它当作正式可切歌曲

这是最务实的路线。

## 数据模型落地

### `Song` 层

`Song` 不直接声明“必定可切换”，而是通过资产集合推导能力。

可以增加一个派生能力字段：

- `capabilities.canSwitchVocalMode`

它不是人工主填字段，而是根据资产情况计算得出。

### `Asset` 层

每个 `Asset` 应明确：

- `vocalMode`：`original / instrumental / dual / unknown`
- `groupKey`
- `switchFamily`

建议语义：

- `groupKey`：标识这是不是同一版本族的资源
- `switchFamily`：标识哪些资源理论上可以互切

例如：

- `original.mp4` 和 `instrumental.mp4` 都属于 `switchFamily = "jaychou-qilixiang-v1"`

这样系统就知道：

- 它们是同一首歌的可切换配对资源
- 而不是两个互不相干的视频版本

### 为什么需要 `switchFamily`

因为同一首歌可能有很多资产：

- MV 原唱版
- 伴奏字幕版
- 演唱会版
- 旧版伴奏

不是所有 `original` 和 `instrumental` 都应该互切。

所以必须有一个显式机制告诉系统：

- 哪些资源是“同一版本族的可切换对”

## 第一版切换规则

### 规则 1：只有成对资源才显示切换按钮

若当前歌曲存在：

- 一个 `original`
- 一个同 `switchFamily` 的 `instrumental`

才在手机端展示“原唱 / 伴唱切换”。

否则：

- 不显示
- 或显示为不可用

### 规则 2：切换本质上是“切换目标资产”

第一版不要把它理解成“在同一播放器实例里热切音轨”。

第一版应理解为：

- 用户请求切换
- 服务端选择同族备用 `Asset`
- 电视端切换到新的目标资源

这是一个“状态机切换资源”的过程。

### 规则 3：播放中切换是正式歌库主路径能力

第一版不再把“播放前选择版本”当成切换能力的替代品。

更准确的理解是：

- 点歌前仍然可以选默认资源版本
- 但正式歌库主路径必须支持播放过程中从当前资源切到同族目标资源
- 切换体验目标是接近无感，至少不能出现明显重新开头、明显黑屏或明显时间轴跳变

推荐运行时流程：

1. 服务端记录当前播放进度和当前 `activeAssetId`
2. 服务端选择同 `switchFamily` 的目标 `Asset`
3. TV 端预加载目标资源并尝试从相近时间点重入
4. 成功后再提交新的 `activeAssetId` / `vocalMode` 为当前真相

这件事之所以能成立，前提不是“播放器足够聪明”，而是资源准入足够严格。

### 规则 4：播放中切换必须建立在“同进度重入 + 严格准入”之上

播放中切换只适用于：

- 两个资源属于同一 `switchFamily`
- 两个资源被验证为同版本、同时轴
- 两个资源时长差与对齐误差都在可接受范围内

如果做不到这些前提，问题不应该甩给播放器，而应该在歌库准入阶段直接拦下。

## 资产命名与目录建议

在正式歌库目录下，建议直接用角色名表达：

- `original.mp4`
- `instrumental.mp4`

如存在更多版本，可扩展为：

- `original-live.mp4`
- `instrumental-live.mp4`
- `original-mv.mp4`
- `instrumental-mv.mp4`

但注意：

- 文件名只是辅助
- 最终是否可切换，仍以 `song.json.assets[].switchFamily` 为准

## 在线歌源与原唱 / 伴唱

在线候选要支持原唱 / 伴唱切换，比本地歌库更难。

第一版建议：

- 在线候选默认只缓存并播放“一个明确版本”
- 不强制在线补歌必须同时找到原唱版和伴唱版
- 若在线源恰好存在成对资源，可作为增强能力接入

原因：

- 在线平台元数据通常不规范
- 很难可靠判断两个资源是否属于同一切换族
- 过度自动化会严重污染切换体验

## UI 与交互建议

### 手机端

建议把“原唱 / 伴唱”理解为一个资源选择器，而不是一个神奇按钮。

第一版可支持：

- 点歌时选择默认进入原唱版或伴唱版
- 若当前歌曲满足切换前提，播放中展示切换入口
- 未成对资源时不展示切换入口

### 电视端

电视端只负责显示当前实际播放的是：

- 原唱
- 伴唱

不负责判断“能不能切”。

### 状态提示

建议在控制端显式展示：

- `仅原唱`
- `仅伴唱`
- `可切换`

这样用户不会误以为所有歌都支持切换。

## 失败与降级策略

### 情况 1：切换目标资源缺失

处理方式：

- 不显示切换能力
- 或切换请求直接返回不可用

### 情况 2：切换目标资源损坏

处理方式：

- 服务端拒绝切换
- 保持当前资源继续播放
- 记录 `PlaybackEvent`

### 情况 3：播放中切换失败

播放中切换失败是第一版必须处理的正式场景。

处理方式：

- 立即回退到切换前资源或维持切换前播放状态
- 给用户展示产品化提示，而不是裸技术错误
- 将这对资源标记为异常待复核
- 不允许因为切换失败直接把整首歌打断到不可恢复

## 原唱 / 伴唱策略的关键取舍

1. 第一版优先“双视频资源切换”，不优先“双音轨”  
   因为它最稳定、最符合当前歌库路线。

2. 第一版把切换理解为“换资源”，不是“变魔术”  
   这样状态机、播放器和歌库模型都更一致。

3. 没有成对资源就明确降级  
   不伪造切换能力，不做运行时智能分离。

4. 播放中切换是第一版目标，但前提必须被严格收紧  
   不靠播放器“猜着切”，而靠成对资源准入、预加载、同进度重入与失败回退保证体验。

## 升降调策略

升降调是很典型的 KTV 功能，但它和“硬件混响”不同，基本不可完全交给外部 mixer 解决。

它作用在歌曲伴奏本身，因此实现位置必须落在软件播放链路里。

这里先定一个总原则：

- 第一版不做实时、任意步进、播放中可随时切换的升降调
- 架构上预留 `pitchSemitones`，后续按“受控转码资产”方式实现

原因不是它不重要，而是它会直接把首版播放器复杂度和稳定性风险拉高。

## 为什么升降调比看上去复杂

如果只是“让声音变高或变低”，最简单的方法是改 `playbackRate`。

但这会同时改变：

- 音高
- 节奏 / 时长

这对 KTV 来说通常不可接受，因为：

- 歌会变快或变慢
- 视频与字幕体验会被破坏
- 切换时很不自然

真正可用的 KTV 升降调，本质上要做的是：

- 改变音高
- 尽量保持时长和节奏不变

这就进入了音频 DSP / 转码范畴。

## 可行实现路径对比

### 方案 1：浏览器端实时升降调

在电视端前端里直接做 pitch shift。

优点：

- 交互最直接
- 理论上可做到即时切换

缺点：

- 浏览器实现复杂
- 电视端性能和兼容性风险高
- 音质容易劣化
- 很难保证和视频播放稳定共存

结论：

- 第一版不建议采用

### 方案 2：服务端按需转码生成临时升降调资源

当用户选择 `+1`、`-2` 等调值时，服务端用 `ffmpeg` 或等价管线生成一个新的临时资源。

优点：

- 电视端播放器逻辑简单
- 更适合你现有家庭服务器架构
- 能把复杂度压在服务端工作队列里

缺点：

- 首次等待时间增加
- 要管理临时缓存生命周期
- 转码质量和速度要评估

结论：

- 后续增强推荐主方案

### 方案 3：预生成多调版本

给热门歌曲预先生成：

- `-2`
- `-1`
- `0`
- `+1`
- `+2`

等多个版本。

优点：

- 播放时最快
- 对电视端最简单

缺点：

- 存储占用明显增加
- 歌库维护成本高
- 对全库不现实

结论：

- 适合作为热门歌缓存优化，不适合作为第一版基础策略

## 第一版建议结论

第一版建议：

- `升降调功能不纳入 MVP`
- 只在数据模型和命令协议里预留能力位

也就是：

- 可以记住用户想要的目标调值
- 但第一版默认不真正执行音频升降调

这样做的好处是：

- 不破坏当前“硬字幕视频 + 稳定播放器”的主路线
- 不把 TV Player 变成 DSP 实验场
- 不影响点歌、搜索、切歌这些首版关键路径

## 为什么第一版先不做

### 1. 你当前主路线是硬字幕视频

这意味着第一版大量资源会是：

- 已编码好的视频
- 歌词直接烧录在画面里

一旦做升降调，就会出现两个问题：

- 只改音频，不改视频，必须做分离处理
- 改完后还要重新封装成可稳定播放资源

这已经不是一个简单播放器开关。

### 2. 电视端浏览器不适合承担这类实时 DSP

即便在桌面浏览器能做，也不代表在电视端或盒子端稳定。

对家庭 KTV 来说，最重要的是：

- 点了就播
- 不黑屏
- 不爆音
- 不卡顿

首版不值得为了升降调牺牲这些基础体验。

### 3. 在线补歌会被进一步复杂化

如果在线候选也要支持升降调，就意味着：

- 候选缓存后还要再派生一层变调缓存
- 资源生命周期更复杂
- 失败恢复更难

所以升降调应在主播放链稳定之后再接入。

## 数据模型预留

虽然第一版不执行升降调，但建议从现在开始预留字段。

### `QueueEntry.playbackOptions`

建议补充：

- `pitchSemitones`

语义：

- 整数或半整数调值
- `0` 表示不变调

第一版即便不执行，也可以：

- 保存用户意图
- 为以后协议兼容保留空间

### `Asset`

后续若做服务端转码，可增加：

- `derivedFromAssetId`
- `transformType`
- `transformParams`

例如：

- `transformType = pitch_shift`
- `transformParams = { "semitones": -2 }`

这样可以把变调后的临时资源也纳入统一 `Asset` 模型。

## 后续增强推荐路径

如果未来要做升降调，推荐按以下顺序演进。

### 阶段 1：播放前选择调值

用户在点歌前选择：

- `-2`
- `-1`
- `0`
- `+1`
- `+2`

系统流程：

1. 服务端收到点歌命令和 `pitchSemitones`
2. 若存在对应派生缓存，直接复用
3. 若不存在，则进入后台转码
4. 转码完成后生成临时 `ready Asset`
5. 再进入正常播放流程

这是我推荐的第一条可实施路线。

### 阶段 2：热门歌曲缓存常用调值

对于播放频率高的歌曲：

- 可以保留若干常用调值缓存
- 避免每次临时转码

### 阶段 3：播放中切换调值

这是最难的一步。

若以后真要做，需要解决：

- 当前播放进度迁移
- 新资源切入平滑度
- 黑屏 / 音画跳变
- 缓存命中率

这明显不适合第一版。

## UI 与交互建议

### 第一版

第一版控制端可暂时：

- 不展示升降调
- 或展示为“即将支持”

建议不要做一个能点但无真实效果的假功能。

### 后续增强

若进入后续阶段，建议把升降调放在：

- 点歌确认页
- 当前歌曲设置页

但默认仍应限制可选范围，例如：

- `-2` 到 `+2`

不要一开始开放过大的自由范围。

## 与原唱 / 伴唱的关系

升降调和原唱 / 伴唱是两条不同能力线。

建议优先级：

1. 先把原唱 / 伴唱资源切换做稳
2. 再考虑对“伴唱版资源”做变调派生

原因：

- 伴唱才是升降调的主使用场景
- 先对结构清晰的 `instrumental Asset` 做变调，复杂度更可控

## 升降调策略的关键取舍

1. 第一版不把升降调放进 MVP  
   先保稳定，再做高级音频能力。

2. 不走浏览器端实时 DSP 主路线  
   电视端不应承担这类高风险计算。

3. 后续优先走“服务端按需转码生成临时资产”  
   让复杂度留在服务端，而不是播放器端。

4. 先支持“播放前选调”，再考虑“播放中切调”  
   这样可以最大限度压低状态机复杂度。

## 数据库 Schema 与索引落地

前面已经定义了核心模型，这里把它收敛成可实现的关系型结构。

目标：

- 第一版结构够稳，不被后续功能轻易推倒
- 搜索与队列主链路性能可控
- 不为了未来可能性过度拆表

## 索引策略建议

### 搜索相关索引

第一版搜索主路径建议依赖：

- `pg_trgm`
- btree 普通索引

推荐启用：

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
```

建议索引：

- `songs.normalized_title` trigram index
- `songs.title_pinyin` trigram index
- `songs.title_initials` btree or trigram index
- `artists.normalized_name` trigram index
- `artists.name_pinyin` trigram index
- `song_aliases.normalized_alias_text` trigram index
- `artist_aliases.normalized_alias_text` trigram index

原因：

- 中文模糊搜、拼音搜、别名搜都需要不同索引支撑

### 队列与状态索引

建议索引：

- `queue_entries(room_id, status, queue_position)`
- `queue_entries(room_id, created_at desc)`
- `playback_events(room_id, created_at desc)`
- `device_sessions(room_id, last_seen_at desc)`

原因：

- 当前房间队列和状态是高频读
- 事件是高频写加排障读

### 资源查找索引

建议索引：

- `assets(song_id, status)`
- `assets(song_id, vocal_mode, switch_family)`
- `assets(checksum)`
- `source_records(provider, provider_song_id)`

原因：

- 点歌时要快速找默认资源
- 原唱 / 伴唱切换要快速找同族资源
- 在线候选缓存要快速避免重复

## 搜索实现建议

第一版不要急着引入独立搜索引擎。

建议策略：

1. 主表走 PostgreSQL
2. 规范化字段、拼音字段、别名表共同参与搜索
3. 服务端聚合计算排序分数

### 查询思路

可按以下层次组合：

1. 标题精确命中
2. 歌手精确命中
3. 标题规范化模糊命中
4. 拼音全拼命中
5. 首字母命中
6. 别名命中

再统一按服务端规则排序。

原因：

- 排序逻辑是产品能力，不应该完全交给数据库默认相似度

## JSONB 使用边界

第一版可以使用 `jsonb`，但要有边界。

适合放 `jsonb` 的：

- `playback_options`
- `event_payload`
- `candidate_meta`
- `prepared_fetch`
- `transform_params`
- `capabilities`

不适合放 `jsonb` 的：

- 主搜索字段
- 主状态字段
- 高频筛选字段

原因：

- 核心检索维度必须显式列化，便于建索引和维护

## 唯一性与去重原则

第一版不要过度依赖数据库强唯一。

建议：

- 对“绝对实体标识”做唯一，例如 `rooms.slug`
- 对“可能存在歧义的业务对象”只做辅助索引，不强行唯一

例如：

- `songs` 不强制 `(artist_id, normalized_title)` 唯一
- `assets` 不强制 `(song_id, vocal_mode)` 唯一

原因：

- 家庭 KTV 歌库里版本和命名非常脏
- 先允许并存，再通过业务层与人工校正清理

### 建议同步建好

### 可后补增强

## 数据库设计的关键取舍

1. 第一版优先关系清晰，不优先“最范式化”  
   让工程落地快，后续再收紧。

2. 搜索能力先靠 PostgreSQL 字段设计和索引设计撑起来

4. 在线候选与正式队列分离

## 二维码入场与设备绑定策略

你的使用场景是：

- 单房间
- 单电视
- 手机是唯一控制端

所以这层设计的目标不是做复杂账号系统，而是：

- 让用户在客厅里拿起手机就能快速进入控制页
- 让电视播放器稳定绑定到 `living-room`
- 防止手机误连到错误房间或错误播放器

## 设备角色划分

第一版只定义两类设备：

- `tv-player`
- `mobile-controller`

角色边界：

### `tv-player`

职责：

- 固定绑定到某个 `room`
- 长时间在线
- 负责播放与状态上报

特点：

- 它不是临时访客设备
- 它更像房间基础设施的一部分

### `mobile-controller`

职责：

- 扫码进入控制页
- 发起点歌与控制命令
- 查看当前状态

特点：

- 可短时加入
- 可有多个同时在线
- 默认不需要复杂登录

## 第一版绑定策略

第一版建议采用：

- `电视端强绑定`
- `手机端轻绑定`

具体含义：

### 电视端强绑定

电视播放器第一次部署时，显式绑定到：

- `room = living-room`

后续正常运行时：

- 不需要每次重新扫码
- 不需要用户反复确认

可以通过：

- 本地配置文件
- 首次配对令牌
- 固定播放器 URL

完成绑定。

建议第一版最简单方式：

- 电视端使用固定 URL，例如 `/player/living-room`
- 服务端在设备首次连接时下发并保存 `deviceId`

### 手机端轻绑定

手机端每次通过二维码进入：

- 房间控制页
- 自动附带 `roomSlug`
- 自动附带短时 `pairingToken`

手机不需要变成“永久设备”，只需要：

- 能进入对的房间
- 能拿到当前状态
- 能发送控制命令

## 二维码内容建议

第一版二维码不建议只编码一个纯页面 URL。

建议包含最小必要上下文，例如：

```text
https://ktv.shaolongfei.com/controller/living-room?token=xxxx
```

推荐包含：

- `roomSlug`
- `pairingToken`
- 可选 `expiresAt` 信息由服务端校验

原因：

- 只靠 `roomSlug` 容易被长期复用
- 加短时 `token` 更适合客厅临时接入

## `pairingToken` 设计

第一版 `pairingToken` 不需要太复杂，但要满足：

- 短期有效
- 可撤销
- 只作用于一个房间

建议语义：

- token 绑定到 `room_id`
- token 有过期时间
- token 可被轮换

建议存储字段：

- `token`
- `room_id`
- `issued_by`
- `expires_at`
- `revoked_at`
- `usage_limit`
- `used_count`

第一版若不单独建表，也可以先挂在 `device_sessions` 或 `rooms` 相关记录上，但长期看更适合独立成 `pairing_tokens` 表。

## 第一版推荐流程

### 电视端首次部署

1. 电视打开固定播放器地址
2. 服务端识别这是 `tv-player`
3. 若该播放器尚未登记，生成 `deviceId`
4. 绑定到 `living-room`
5. 后续用 `deviceId` 做稳定识别

第一版由于是单房间，可简化为：

- 只允许一个主 TV Player
- 若已有主播放器在线，新播放器连接时提示冲突

### 手机端入场

1. 电视端画面上始终展示或按需展示二维码
2. 手机扫码后进入控制端 URL
3. 控制端携带 `roomSlug + pairingToken`
4. 服务端验证 token
5. 验证通过后建立手机控制会话
6. 返回当前完整状态

### 手机端控制会话

会话建立后：

- 通过 WebSocket 接收实时状态
- 通过 HTTP 或 WebSocket 发送控制命令

建议：

- 控制会话可短期保活
- 即使二维码过期，已建立的会话可继续使用一段时间

这样用户不会因为唱到一半 token 过期就掉线。

## 二维码展示策略

### 方案 1：始终展示小二维码

优点：

- 拿起手机就能扫
- 体验直观

缺点：

- 会占用部分播放画面

### 方案 2：空闲时大二维码，播放时角落小二维码

这是我更推荐的方案。

建议：

- `idle` 状态下展示大二维码
- `playing` 状态下缩成右上角或右下角小码

优点：

- 空闲时便于入场
- 播放时不太打扰画面

### 方案 3：仅在遥控或按键触发时显示

不适合作为第一版默认。

原因：

- 你的目标是手机是唯一控制端
- 如果连进入控制页都还要先用遥控，流程反而绕了

## 多手机同时接入策略

第一版允许多个手机同时在线，但要明确行为规则。

建议：

- 多个手机都可以查看状态
- 多个手机都可以点歌
- 写命令统一走 `expectedVersion`

这样做的结果是：

- 可以多人同时点歌
- 冲突由服务端版本机制处理

### 是否需要房主权限

第一版建议：

- 不做完整权限体系
- 默认所有进入控制页的手机都可点歌、切歌、删歌

如担心误操作，可后续增强：

- `controller` 模式：只能点歌
- `admin` 模式：可切歌、删歌、调整顺序

但这不是第一版必须项。

## TV Player 冲突处理

第一版必须回答一个问题：

- 如果两个 TV Player 同时连到同一个房间怎么办？

建议策略：

1. 每个房间默认只有一个 `active tv-player`
2. 新 TV Player 连接时：
   - 若当前没有活动播放器，则接管
   - 若已有活动播放器在线，则进入 `conflict` 提示态
3. 是否允许抢占，留作后续增强

第一版最简单做法：

- 不允许自动抢占
- 直接提示“该房间已有播放器在线”

## 控制端会话与断线恢复

手机控制页不应把二维码 token 当成唯一长期凭证。

建议分两层：

### 入场层

使用：

- `roomSlug`
- `pairingToken`

作用：

- 进入房间
- 建立控制会话

### 会话层

建立成功后，服务端发放：

- `controllerSessionId`
- 短期会话 cookie 或 token

作用：

- 后续命令鉴权
- 页面刷新后短时恢复

这样设计的好处是：

- 二维码可以短期有效
- 会话可以比二维码寿命更长
- 既方便又不至于太裸奔

## 第一版推荐安全边界

这是家庭内网或家庭受控环境产品，不需要一开始做成企业级安全系统。

但至少要做到：

1. 二维码 token 有过期时间  
   不要永久可用。

2. token 与房间绑定  
   不要一个 token 通吃所有房间。

3. 会话和二维码解耦  
   扫码只负责进入，会话单独维护。

4. 不依赖前端自报房间身份  
   服务端根据 token 和会话确认房间归属。

## 数据库落地建议

为了支持这套设计，建议新增一张表：

- `pairing_tokens`

建议字段：

- `id`
- `room_id`
- `token`
- `issued_for`
- `expires_at`
- `revoked_at`
- `usage_limit`
- `used_count`
- `created_at`

说明：

- `issued_for` 可标记 `controller-entry`、`tv-bootstrap` 等用途
- 第一版主要使用 `controller-entry`

索引建议：

- `token` 唯一
- `(room_id, expires_at)`

## 二维码与设备绑定的关键取舍

1. 电视端固定绑定，手机端临时进入  
   这最符合你的客厅单电视场景。

2. 二维码是“入场票”，不是长期身份  
   扫码后应建立独立控制会话。

3. 第一版允许多手机控制，不先做复杂权限  
   把复杂度留给后续，而不是一开始加账号系统。

4. 房间和播放器必须强绑定  
   避免未来扩展时出现串房和串播。

## 具体 API Schema 与事件 Payload

这一层的目标是把前面已经确认的：

- HTTP API
- WebSocket 协议
- 状态机

收敛成可直接映射到 `Zod` / TypeScript 类型的合同。

原则：

- 第一版追求协议清晰，不追求字段极简
- 优先让服务端和两个前端共享同一套 schema
- 所有消息都采用稳定 envelope

## 通用字段约定

### ID 类型

第一版建议所有主标识统一用字符串 UUID 或等价长随机 ID：

- `artistId`
- `songId`
- `assetId`
- `queueEntryId`
- `roomId`
- `deviceId`
- `candidateTaskId`

原因：

- 避免前后端耦合数据库自增 ID 语义
- 后续更容易迁移和外露接口

### 时间字段

统一使用 ISO 8601 字符串：

- `createdAt`
- `updatedAt`
- `startedAt`
- `endedAt`
- `expiresAt`

### 版本字段

所有与房间真相强相关的写操作和广播消息都带：

- `version`

用于：

- 客户端状态同步
- 并发冲突控制
- 断线恢复判断

## Schema 设计的关键取舍

1. 第一版优先显式字段，不优先极限抽象  
   便于调试、前后端协作和日志排障。

2. 所有关键写命令带 `expectedVersion`  
   多手机并发控制必须有基础约束。

3. 服务端广播优先“全量可恢复”  
   不让客户端依赖脆弱的局部 patch 历史。

4. 在线候选状态单独建模  
   不把异步缓存流程塞进正常队列协议里。

## 后台管理能力范围

这套系统第一版确实需要后台，但它应该是：

- 面向系统维护者的运营后台
- 不是面向普通唱歌用户的复杂管理平台

也就是说，后台的目标不是“做全”，而是：

- 让歌库可维护
- 让播放器和房间状态可排障
- 让在线补歌和导入流程可控

## 后台的角色定位

第一版后台只服务于一个主要角色：

- `system-admin`

这个角色默认就是你自己。

第一版不建议单独引入复杂 RBAC，但可以预留简单角色概念：

- `viewer`
- `operator`
- `admin`

实际第一版可只实现：

- 一个受保护的后台入口
- 一个管理员身份

## 第一版后台必须包含的能力

### 1. 歌库管理

这是后台第一优先级。

应至少支持：

- 浏览歌曲列表
- 查看歌曲详情
- 查看同一首歌的所有 `Asset`
- 查看 `lyricMode`、`vocalMode`、`switchFamily`
- 修改歌曲元数据
- 修改歌手元数据
- 调整 `defaultAsset`
- 标记歌曲状态：`ready / incomplete / review_required`

原因：

- 前面设计里已经明确，歌库不会天然干净
- 没有后台的话，后期只能直接改数据库或文件

### 2. 导入审核

应至少支持：

- 查看 `imports/needs-review` 候选
- 查看扫描器推断结果
- 人工确认歌手 / 歌名 / 语种
- 手工指定 `vocalMode`
- 决定：
  - 正式入库
  - 继续搁置
  - 拒绝导入

原因：

- 你的系统主路径是“本地歌库 + 人工校正”
- 这个环节没有后台就很难运维

### 3. 在线候选与缓存任务管理

应至少支持：

- 查看 `candidate_tasks`
- 查看当前状态：`selected / fetching / fetched / ready / failed`
- 查看失败原因
- 手工重试
- 手工清理失败任务
- 将合适的在线缓存资源“转正”为正式歌曲

原因：

- 在线补歌是最不稳定的一层
- 必须有一个人工接管点

### 4. 房间与播放器状态查看

应至少支持：

- 查看当前房间状态
- 查看 `PlaybackSession`
- 查看当前 `active tv-player`
- 查看最近心跳时间
- 查看当前队列
- 查看最近 `PlaybackEvent`

原因：

- 这能让你快速判断问题是在：
  - 播放器端
  - 服务端状态机
  - 资源层

### 5. 二维码与配对管理

应至少支持：

- 查看当前 `pairingToken`
- 手工刷新二维码
- 让旧 token 失效
- 查看当前在线控制端数量

原因：

- 这是实际客厅使用流程的一部分
- 也是最容易排查“为什么手机进不去”的入口

## 第一版后台建议包含但可简化的能力

### 1. 搜索调试视图

建议有一个简化页面，用于查看：

- 某首歌的索引字段
- `normalized_title`
- `title_pinyin`
- `title_initials`
- `aliases`
- `searchHints`

原因：

- 中文搜索调优时非常有用

### 2. 资源健康检查

建议可查看：

- 文件是否存在
- `ffprobe` 是否成功
- 资源状态是否为 `ready`
- 是否缺歌词

这块第一版不必做得太花，但要能看。

### 3. 操作历史

建议最少提供最近操作日志：

- 谁点了歌
- 谁删了歌
- 谁跳了歌
- 谁转正了在线资源
- 谁修改了歌曲元数据

如果第一版不做完整审计页，至少应能从 `playback_events` 和后台操作日志中看到。

## 第一版后台明确不做

### 1. 复杂权限系统

不做：

- 多级角色
- 精细到按钮级别的权限控制
- 多人协同审批

### 2. 复杂运营内容系统

不做：

- 海报运营位
- 推荐策略后台
- 首页编排系统
- 复杂标签策展台

### 3. 全功能媒体编辑器

不做：

- 浏览器内剪视频
- 浏览器内做字幕编辑
- 浏览器内做音轨处理

这些都不是第一版需要承受的复杂度。

第一版建议后台导航尽量少，控制在 5 到 7 个主页面内。

建议主导航：

- `Overview`
- `Songs`
- `Imports`
- `Online Tasks`
- `Room`

### `Overview`

展示：

- 在线失败任务数
- 当前房间状态

### `Songs`

展示：

- 歌曲列表
- 搜索

  - `language`
  - `status`
  - `hasSwitchPair`
  - `lyricMode`

### `Imports`

展示：

- `pending`
- `needs-review`
- `rejected`

### `Online Tasks`

展示：

- 当前和历史 `candidate_tasks`

### `Room`

展示：

- 当前房间状态

- 当前队列
- 最近事件

### `Devices`

展示：

- TV Player
- 当前在线 mobile controllers

### `Settings`

第一版只建议放少量系统级设置，例如：

## 后台与主用户端的关系

要明确：

- 后台不是给唱歌用户用的
- 主控制端保持极简和面向唱歌流程
- 一切“修数据”“修资源”“看失败任务”的事都进后台

这能避免把手机控制端做成半个管理系统。

## 实现建议

### 1. 后台与控制端共用同一套 API

后台应优先复用：

- 歌曲查询接口
- 房间状态接口
- 设备状态接口

必要时再增加后台专用接口。

原因：

- 避免两套平行业务逻辑

### 2. 后台需要少量专用接口

例如：

- 更新歌曲元数据
- 批准导入
- 拒绝导入
- 重试在线任务
- 转正在线资源
- 刷新 pairing token

这些接口不应暴露给普通控制端。

### 3. 后台要优先可用性，不优先炫酷

第一版后台核心要求是：

- 快速查问题
- 快速改数据
- 快速推进导入流程

而不是视觉上多华丽。

## 后台设计的关键取舍

1. 第一版后台做“运营与运维控制面板”，不做“产品平台”  
   重点是歌库维护和系统排障。

2. 后台必须覆盖导入审核和在线任务  
   这两条链路如果没后台，后面维护成本会非常高。

3. 后台与主控制端彻底分工  
   手机端管唱歌，后台管维护。

4. 后台功能宁可少而硬，不要多而散  
   能解决主问题最重要。

## 在线歌源接入方式与合规边界

这一部分需要同时解决两个问题：

1. 技术上怎么接
2. 边界上哪些事不做

这里先给出总原则：

- 这套系统的主路径永远是本地歌库
- 在线歌源只做补充，不做基础依赖
- 第一版不设计成“自动抓取并长期沉淀第三方平台内容”的系统

## 风险分层

从合规和技术稳定性看，在线歌源大致可分成 3 档：

### 1. 自有或已获授权内容

这是风险最低、最干净的来源。

例如：

- 你自己拥有使用权的伴奏视频
- 明确授权可下载、缓存、播放的资源
- 你自行制作并导入的 KTV 资源

结论：

- 可以作为正式歌库主来源

### 2. 平台官方允许的“站内访问 / 嵌入 / API 数据”

这类来源通常允许：

- 搜索元数据
- 站内播放或嵌入播放

但不等于允许：

- 下载音视频
- 长期缓存音视频
- 脱离平台独立播放

结论：

- 可以做“发现层”或“跳转层”
- 不能默认当成“可抓取并沉淀进本地歌库”的来源

### 3. 非官方抓取、下载、解析

这类方式技术上常常可行，但风险最高。

风险来自：

- 平台服务条款
- 版权与邻接权
- 接口稳定性
- 账号与 IP 风险

结论：

- 不应作为第一版正式产品能力写进主架构

## 基于官方资料的边界判断

### YouTube

根据我在 2026-04-28 查到的官方开发者政策和服务条款，YouTube 的 API 客户端不得在未获得书面批准的情况下下载、导入、备份、缓存或存储 YouTube 音视频内容，也不得提供离线播放。YouTube 服务条款也把用户对内容的使用限制在“通过服务启用的功能”内，而不是脱离服务独立使用。

这意味着：

- 可用 YouTube 官方 API 做搜索、元数据、嵌入式播放等合规能力
- 不应把 YouTube 作为“自动下载并缓存到本地 KTV 歌库”的官方合规路径

### Bilibili

我没有在这次查阅的官方资料中找到一个面向这类家庭 KTV 产品的、公开明确授权“下载并缓存站内音视频内容到自建系统离线播放”的官方开发路径。

这不等于绝对禁止，但在没有明确官方授权路径前，不能把它当作默认可行的正式产品能力。

因此更稳妥的结论是：

- 不把 Bilibili 等平台视为“可默认抓取沉淀”的正式在线源
- 若未来要接，必须先确认具体授权、平台规则和使用边界

## 推荐的在线接入分层

为了既保留扩展能力，又不把系统架构建在高风险路径上，建议把在线能力拆成 3 层：

### 层 1：Discovery（发现层）

系统可做：

- 搜索在线候选
- 展示标题、歌手、缩略图、来源平台
- 告知用户这是“在线候选”，不是正式本地歌

这一层主要作用是：

- 帮助用户知道外部平台上是否可能有这首歌

### 层 2：Operator-mediated Import（人工主导导入）

系统可做：

- 记录候选
- 后台人工审核
- 由管理员确认是否可以导入
- 仅对明确有权使用的资源执行正式导入

这是我对你当前系统最推荐的中间态。

### 层 3：Platform-compliant Playback（平台合规播放）

如果未来某个平台有明确允许的嵌入或官方播放方式，可单独做：

- 在原平台能力边界内播放
- 不把音视频沉淀到本地歌库

但这不应该和“本地正式歌库播放链”混成一套。

## 第一版建议结论

第一版对在线歌源的正式策略建议如下：

### 允许做的

- 在线候选发现
- 在线候选元数据展示
- 记录候选来源
- 后台人工审核
- 对你明确有权使用的资源执行导入

### 不建议做进主产品能力的

- 自动批量抓取第三方平台内容
- 默认缓存第三方平台音视频到本地
- 默认把第三方平台内容转正为正式歌库
- 绕开平台官方能力边界做“伪离线”

## 架构落地建议

### 1. `provider` 必须是可替换适配器

在线源在技术上应仍然保留：

- `search`
- `getCandidate`
- `prepareFetch`
- `verify`

但是否真正启用 `prepareFetch`，取决于该来源是否有清晰的授权路径。

换句话说：

- 不是所有 provider 都必须支持“抓取到本地”
- 有些 provider 只支持 Discovery

### 2. 给 `provider` 增加能力声明

建议为每个在线源适配器增加能力位：

- `canSearch`
- `canEmbed`
- `canImport`
- `canCache`
- `requiresManualReview`

这样业务层可以明确知道：

- 哪些来源只能搜
- 哪些来源可以人工导入
- 哪些来源具备明确缓存授权

### 3. 候选状态机增加“人工审核闸门”

对于存在合规不确定性的来源，建议：

- `selected -> review_required -> approved -> fetching`

而不是直接：

- `selected -> fetching`

这能把风险前置到后台人工确认层。

### 4. “转正”必须附带来源说明

当某个在线资源被转正为正式歌库时，后台应要求填写或保留：

- 来源平台
- 导入方式
- 权限说明 / 备注

哪怕第一版只是文本备注，也比完全没有记录强很多。

## 对当前产品路线的最终建议

结合你的目标和当前风险，我建议把在线能力定义成：

- 第一版：`在线发现 + 人工审核导入`
- 第二版：只在有明确规则和可控来源时，扩展到更自动化的导入

不要把第一版做成：

- “搜到就自动抓”
- “抓到就自动缓存”
- “缓存后自动沉淀正式歌库”

这是最容易在技术上看似顺手、在长期上最不稳的路线。

## 这部分的关键取舍

1. 在线歌源是发现层优先，不是抓取层优先  
   先解决“找得到”，再解决“能不能合法稳定导入”。

2. 平台规则不明时，默认走人工审核闸门  
   不把不确定性自动化。

3. 正式歌库只沉淀你明确可控制、可维护、可解释的资源  
   这是系统长期健康的底线。

4. 第一版的在线能力要设计成“可关闭”  
   即便完全关闭在线源，本地歌库主流程也必须完整可用。

## 当前待定项

当前这一轮架构设计中，核心待定项已全部收敛。

后续若继续推进，重点将转入实施层工作：

- 将本文转成可执行实现计划
- 落具体表结构、接口类型与页面信息架构
- 再决定是否启动原型或代码仓搭建

## 架构复盘与最终方案

在完整走过这套设计后，当前真正存在“过度设计风险”的点主要有 4 个：

1. 把逻辑模块误当成独立部署服务  
   第一版应该是模块化单体，不应该拆微服务。

2. 为单房间单实例场景过早引入过多基础设施  
   `Redis`、独立搜索引擎、复杂权限系统都不是 MVP 必需项。

3. 为未来预留过多媒体能力  
   `soft_sub`、`external_lrc`、双音轨、升降调，如果一起进主路径，会直接拉高首版复杂度。

4. 把在线补歌设计得太自动  
   这是最容易在稳定性、合规和维护成本上失控的一层。

基于这轮复盘，最终建议把方案收敛成下面这一版。

## 最终推荐架构

### 部署形态

第一版采用：

- `一个后端应用`
- `两个用户前端`
- `一个轻量后台`

即：

- `backend`
- `tv-player`
- `mobile-controller`
- `admin`

其中：

- `backend` 同时承担 API、WebSocket、状态机、搜索、歌库扫描入口
- 若后续需要异步任务，可先在同机增加 `worker`，但不拆成分布式系统

结论：

- 第一版不做微服务
- 第一版不做多节点
- 第一版按单机自托管系统设计

### 基础设施最小集

第一版只保留：

- `PostgreSQL`
- `NAS 文件存储`
- `ffmpeg / ffprobe`

`Redis` 降级为：

- 非 MVP 必须项
- 只有在你确认需要更强任务队列或 pub/sub 后再引入

### 媒体能力最小集

第一版正式可唱资源默认标准直接收紧为：

- 本地歌库
- 视频型资源
- 硬字幕
- 浏览器友好编码格式，优先 `H.264 + AAC`

这意味着：

- `hard_sub` 是主路径
- `soft_sub`、`external_lrc` 只保留数据模型
- `none` 默认不进入正式歌库

### 原唱 / 伴唱能力边界

第一版只支持：

- 在一对已验证资源之间进行播放中原唱 / 伴唱切换
- 点歌前选择默认起播版本

建立方式：

- 两个明确 `Asset`
- 同一 `switchFamily`
- 接近同时间轴的进度重入

第一版不支持：

- 双音轨优先方案
- 软件人声消除
- 未经验证资源对的播放中切换

### 在线能力最终收口

第一版在线能力定义为：

- 默认可关闭
- 若开启，只做 `在线发现 + 后台人工审核导入`

第一版不做：

- 自动抓取
- 自动缓存
- 自动转正
- 在线直通播放主路径

### 升降调最终收口

第一版不做升降调执行能力。

只保留：

- `pitchSemitones` 字段

后续若要做，走：

- 服务端按需转码派生临时 `Asset`

### 二维码与设备绑定最终收口

第一版采用：

- TV 固定绑定 `living-room`
- 手机扫码临时进入控制页

进一步简化建议：

- `pairingToken` 可以先做成短时签名 token
- 若后续需要撤销、审计或多房间扩展，再落 `pairing_tokens` 表

### 搜索能力最终收口

第一版搜索只依赖：

- PostgreSQL
- 规范化字段
- 拼音字段
- 别名表
- `pg_trgm`

第一版不引入：

- Elasticsearch
- Meilisearch

### 后台能力最终收口

第一版后台只做 5 件事：

- 歌库管理
- 导入审核
- 在线任务查看/处理
- 房间与播放器状态排障
- 二维码/设备管理

不做：

- 复杂 CMS
- 复杂 RBAC
- 浏览器内媒体编辑

## 最终 MVP 范围

如果把第一版压到真正可落地、风险最低，MVP 只交付以下能力：

1. 本地歌库扫描与入库
2. 中文搜索
3. 手机点歌
4. 已点队列
5. 切歌 / 删歌 / 顶歌
6. TV Player 全屏稳定播放
7. 二维码入场
8. 原唱 / 伴唱播放中切换（仅限已验证同族资源对）
9. 后台导入审核与歌库维护

MVP 不包含：

1. 升降调
2. 在线自动抓取与自动转正
3. `external_lrc` 重体验
4. 多房间
5. 复杂权限系统

## 最终技术方案

把前面的设计压成一句话，最终建议是：

`单机单实例后端 + PostgreSQL + NAS 本地硬字幕视频歌库 + Web TV Player（按未来 Android TV 壳边界设计）+ 手机控制端 + 轻量后台`

这是目前最稳、最符合你真实使用场景、也最不容易因为过度设计而拖慢实现的方案。


## 当前结论

当前方向已经明确：

- 产品形态选择“家庭包厢式 KTV”
- 架构采用“服务端状态机 + 电视播放器 + 手机控制器”
- 歌源采用“本地主、在线辅”的混合模式
- 音频效果处理交给硬件链路
- 第一版优先解决点歌、搜索、播放、队列和稳定性

后续设计应围绕“先做稳定可唱，再做高级体验”推进。
