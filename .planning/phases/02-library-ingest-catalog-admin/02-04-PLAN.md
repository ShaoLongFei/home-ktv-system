Build `ImportWorkbench.tsx`:
- Fetch candidate queues for `pending`, `held`, `review_required`, and `conflict`.
- Show `conflict` as a first-class filter/status alongside pending, held, and review-required.
- Provide a manual scan button.
- Render a dense grouped list per D-06 with status, artist/title, language, selected file count, and warning indicators.
- Load a selected candidate detail and pass it to `CandidateEditor`.

Build `CandidateEditor.tsx`:
- Provide editable inputs/selects for all D-07 fields.
- Render candidate files with swatches/status badges for root kind and probe status.
- Let admin choose proposed original/instrumental roles and selected files.
- Save ordinary metadata without confirmation.
- Use confirmation for approve and reject-delete; hold can be direct but must show that files go to `imports/needs-review`.
- For candidates with status `conflict`, show conflict metadata (`targetDirectory`, `conflictType`, and matched `songId` if present) without exposing absolute filesystem paths beyond the server-provided value.
- Provide explicit D-02 conflict resolution controls:
  - merge-existing action/form posts to `POST /admin/import-candidates/:candidateId/resolve-conflict` with `{ resolution: "merge_existing", targetSongId }` and requires a non-empty target song id.
  - create-version action/form posts to the same `resolve-conflict` endpoint with `{ resolution: "create_version", versionSuffix }` and requires a non-empty suffix.
  - Do not auto-submit either resolution after a failed approve; the admin must choose one.
- Keep reject/delete confirmation behavior unchanged.

Build `ConfirmActionDialog.tsx` for dangerous actions. Keep styling compact and operational; use icons only if the existing dependencies already provide them, otherwise use text buttons.
