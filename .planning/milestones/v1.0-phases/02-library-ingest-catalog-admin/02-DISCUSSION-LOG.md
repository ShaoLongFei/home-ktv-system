## 导入文件归档方式

### 审核通过后的正式文件位置

| Option | Description | Selected |
|--------|-------------|----------|
| 移动到正式目录 | 审核通过后从 `imports/pending` 或 `imports/needs-review` 移动到 `/songs/<language>/<artist>/<title>/`，正式歌库只有一份主文件。 | yes |
| 复制到正式目录，保留原始文件 | 便于追溯，但占用更多 NAS 空间，也容易留下重复资源。 |  |
| 不移动文件，只在数据库引用原路径 | 实现快，但长期目录会变乱，不利于人工维护和备份。 |  |

**User's choice:** 移动到正式目录
**Notes:** 正式路径作为主文件位置。

### 正式目录冲突处理

| Option | Description | Selected |
|--------|-------------|----------|
| 先生成候选冲突，必须人工确认 | 不自动合并到已有目录，后台提示可能是同一首或同名不同版本，由管理员选择合并或新建版本。 | yes |
| 高置信度自动合并，低置信度人工确认 | 更省事，但自动合并一旦错了会污染正式歌库。 |  |
| 总是新建带后缀目录 | 最安全不覆盖，但长期会产生很多重复目录。 |  |

**User's choice:** 先生成候选冲突，必须人工确认
**Notes:** 同名歌手 / 同名歌曲目录冲突时不自动合并。

### 正式歌库扫描策略

| Option | Description | Selected |
|--------|-------------|----------|
| 信任 `song.json`，只做轻量校验 | 正式目录以 `song.json` 为业务真相，只校验文件存在、可探测、状态是否一致。 | yes |
| 每次都重新推断元数据 | 能发现更多问题，但容易覆盖人工修正。 |  |
| 完全不扫正式目录，只扫导入目录 | 实现简单，但正式文件被手工改动后系统不容易发现。 |  |

**User's choice:** 信任 `song.json`，只做轻量校验
**Notes:** 正式 `/songs` 目录以 `song.json` 为业务真相。

### 导入扫描触发方式

| Option | Description | Selected |
|--------|-------------|----------|
| 后台手动触发扫描 | 管理员点“扫描导入目录”，系统生成候选项；简单、可控。 | partial |
| 定时扫描 | 间隔扫描查漏补缺。 | partial |
| 文件监听实时扫描 | 无感知触发，但 NAS / LXC 环境下需要注意可靠性。 | partial |

**User's choice:** 三者同时存在
**Notes:** 文件监听无感知，定时扫描查漏补缺但间隔长一些，手动触发作为兜底；扫描必须是轻操作。

## 候选审核流程

### 候选审核主视图

| Option | Description | Selected |
|--------|-------------|----------|
| 按候选歌曲分组 | 一个候选项包含原唱 / 伴唱 / 歌词 / 封面等多个文件，适合审核“这是不是一首可入库歌曲”。 |  |
| 按单个文件列表 | 实现简单，但配对和同版本判断割裂。 |  |
| 两层视图 | 默认按候选歌曲分组，必要时展开看原始文件列表。 | yes |

**User's choice:** 两层视图
**Notes:** 默认候选歌曲分组，展开查看每个原始文件和探测结果。

### 审核可修正字段

| Option | Description | Selected |
|--------|-------------|----------|
| 核心字段优先 | 歌名、歌手、语种、原唱 / 伴唱角色、默认资源、是否同一版本族。 |  |
| 完整元数据一起修 | 再加流派、标签、年份、别名、搜索提示等。 | yes |
| 分阶段 | 入库审核只修核心字段；别名、标签、搜索提示等放到正式歌库维护页补充。 |  |

**User's choice:** 完整元数据一起修
**Notes:** 候选审核阶段即可确认完整元数据。

### 搁置语义

| Option | Description | Selected |
|--------|-------------|----------|
| 暂不处理但保留候选 | 文件留在 `imports/needs-review`，候选状态为 `held`，以后可继续编辑和入库。 | yes |
| 移出主审核列表，只在隐藏 / 归档区可见 | 审核页更干净，但可能忘记处理。 |  |
| 要求填写搁置原因和下次提醒时间 | 更严谨，但第一版流程更重。 |  |

**User's choice:** 暂不处理但保留候选
**Notes:** `held` 候选之后仍可继续编辑和入库。

### 拒绝语义

| Option | Description | Selected |
|--------|-------------|----------|
| 移动到 `imports/rejected` 并保留拒绝记录 | 可追溯，也避免下次扫描再次出现。 |  |
| 只标记拒绝，不移动文件 | 实现简单，但导入目录会越来越乱。 |  |
| 直接删除文件 | 最干净，但误拒绝的恢复成本高。 | yes |

**User's choice:** 直接删除文件
**Notes:** 需要明确二次确认，避免误删。

## 正式入库准入规则

### 单个可播放资源

| Option | Description | Selected |
|--------|-------------|----------|
| 允许入正式库，但不进入可点主路径 | 后台可维护，手机点歌搜索先不展示。 |  |
| 允许入正式库，也允许手机点歌 | 更宽松，但不能切原唱 / 伴唱。 |  |
| 不允许正式入库，只能作为候选搁置 | 最严格，但会让很多可唱资源无法被利用。 | yes |

**User's choice:** 不允许正式入库，只能作为候选搁置
**Notes:** 正式入库必须满足可切换资源对前提。

### 原唱 / 伴唱都有但无法证明同版本同时轴

| Option | Description | Selected |
|--------|-------------|----------|
| 进入 `review-required`，不允许正式入库 | 后台保留候选和探测信息，人工确认或替换资源后再入库。 | yes |
| 允许正式入库，但关闭切换能力 | 资源可用性更高，但弱化严格准入。 |  |
| 直接拒绝 | 最干净，但可能误杀接近合格资源。 |  |

**User's choice:** 进入 `review-required`，不允许正式入库
**Notes:** 无法证明同版本、同时轴时不能进入正式库。

### `300ms-1s` 时长差是否可人工转正

| Option | Description | Selected |
|--------|-------------|----------|
| 允许，但必须人工明确确认同版本同时轴 | 系统保留确认记录，确认后可设为 `verified`。 |  |
| 不允许，`>300ms` 一律不能正式入库 | 最严格，切换体验更稳。 | yes |
| 允许自动入库，只提示风险 | 最宽松，不符合严格准入倾向。 |  |

**User's choice:** 不允许，`>300ms` 一律不能正式入库
**Notes:** `300ms-1s` 可保留为 `review-required` 候选，但不能进入正式主路径。

### 默认播放资源

| Option | Description | Selected |
|--------|-------------|----------|
| 默认伴唱版 | 符合 KTV 主场景；原唱版作为播放中可切换资源。 | yes |
| 管理员审核时手动选择 | 更灵活，但每首歌都多一步。 |  |
| 如果有伴唱选伴唱，否则选原唱 | 适合宽松库，但严格成对后实际等同默认伴唱。 |  |

**User's choice:** 默认伴唱版
**Notes:** 正式入库后 `defaultAsset` 指向伴唱版。

## 歌库后台最小界面

### 后台首页优先工作台

| Option | Description | Selected |
|--------|-------------|----------|
| 导入审核工作台优先 | 进入后台先看到待扫描、待审核、搁置、`review-required` 的候选队列。 | yes |
| 正式歌库列表优先 | 更像 CMS，适合维护，但导入流程入口会弱一些。 |  |
| 概览仪表盘优先 | 展示统计数字和状态卡片，但 MVP 不如直接进入工作流高效。 |  |

**User's choice:** 导入审核工作台优先
**Notes:** Phase 2 后台首先服务扫描、候选审核、搁置和 `review-required` 处理。

### 正式歌库维护编辑能力

| Option | Description | Selected |
|--------|-------------|----------|
| 歌曲与资源核心维护 | 改歌名、歌手、语种、完整元数据、默认资源、资源状态、`switchFamily`、`vocalMode`。 | yes |
| 只允许改歌曲元数据，不允许改资源结构 | 安全一些，但配对错误时后台修不了。 |  |
| 完整 CMS 式维护 | 再加批量编辑、封面、歌词、标签管理、历史记录等。 |  |

**User's choice:** 歌曲与资源核心维护
**Notes:** 支持完整元数据、默认资源、资源状态、`switchFamily`、`vocalMode`。

### 后台权限 / 登录

| Option | Description | Selected |
|--------|-------------|----------|
| 做一个简单受保护入口 | 单管理员口令或本地环境配置，不做复杂账号体系。 |  |
| 完全不做登录，只内网访问 | 开发最快，但误操作风险更高。 | yes |
| 做完整管理员账号体系 | 更正规，但超出 v1 无账号体系边界。 |  |

**User's choice:** 完全不做登录，只内网访问
**Notes:** 后台入口不应暴露到公网。

### 危险操作确认强度

| Option | Description | Selected |
|--------|-------------|----------|
| 危险操作二次确认 | 普通编辑直接保存；删除文件、改配对关系、禁用 ready 资源需要明确确认。 | yes |
| 所有保存都二次确认 | 最安全，但维护效率差。 |  |
| 都不确认，依赖管理员谨慎 | 最快，但拒绝直接删文件时风险偏高。 |  |

**User's choice:** 危险操作二次确认
**Notes:** 删除文件、修改 `switchFamily` / 配对关系、禁用 `ready` 资源等危险操作需要二次确认。
